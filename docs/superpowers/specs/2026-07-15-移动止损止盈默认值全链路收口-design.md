# 移动止损止盈默认值全链路收口设计

> **状态：** 方案 A 已获用户批准；本版已吸收第一轮独立审查  
> **范围：** 先完成设计与计划，不在用户审阅计划前修改生产代码  
> **实施原则：** 最小修改、TDD、显式值零变化、每步回归、任一失败立即停止  
> **后续循环：** 本 CRITICAL 修复完成后重新评分；未达 9.5–9.8 时，按独立根因分批设计、实施和验证，不做一次性大改

## 1. 目标

消除 VERA 默认策略中移动止损止盈参数的跨入口分裂，使所有“没有提供具体参数”或参数值为 `None` 的场景统一采用 3.5% 激活、1% 回撤，同时完整保留用户明确配置的任何非 `None` 数值。

本次只统一默认值解析，不改交易算法、不改信号日 T 收盘买入口径、不改退出优先级、不改接口签名和返回结构。

## 2. 权威默认值与决策证据

本设计采用 0.035 / 0.01，不是因为“YAML 恰好这样写”，而是基于以下三重证据：

1. Git 提交 `1900b8f` 在“移动止盈优先级”功能改造中，主动把 `config/default.yaml` 从 0.08 / 0.05 改为 0.035 / 0.01；这是功能提交中的明确变更，不是本轮临时猜测。
2. 当前 `load_stop_config_or_default()` 的容错配置也已对齐 0.035 / 0.01，并有 `tests/test_stop_config_consistency.py` 锁定 YAML 与代码兜底一致。
3. 用户已在本轮明确批准方案 A：以 `config/default.yaml` 的 3.5% / 1% 为权威默认，并要求不破坏用户现有显式配置。

因此，本轮不是重新设计交易参数，而是完成 `1900b8f` 未收干净的全链路同步。

## 3. 当前问题与行为影响

### 3.1 两套默认值

- `config/default.yaml` 与 `load_stop_config_or_default()`：3.5% / 1%。
- Web 页面、`StrategyConfig.get()`、`BacktestEngine.run()`、`run_cached()`、摘要函数缺字段兜底：8% / 5%。

调用者是否经过 YAML 深度合并会改变退出时机，继而改变交易记录、权益曲线、夏普、Calmar、最大回撤和胜率。

### 3.2 与阶梯止盈及硬止损的交互

当前默认优先级是 `trailing_first`，实际执行顺序为：

1. 阶梯止盈先检查；部分卖出后不阻塞后续检查。
2. 移动止损止盈再检查。
3. 硬止损兜底。

因此，3.5% / 1% **不会禁用** 6% / 15% 阶梯止盈：同一根 bar 触及 6% 时，阶梯止盈仍先执行；若随后 Low 又触及移动止损止盈线，剩余仓位才继续退出。

真实变化只发生在依赖旧缺字段兜底的入口：持仓涨到 3.5% 以上、尚未触及 6% 首档、随后从峰值回撤 1% 时，会比旧 8% / 5% 更早退出。这正是 `1900b8f` 已写入默认配置、但其他入口漏同步的策略行为。

### 3.3 历史结果可比性

- 明确保存了移动止损止盈数值的历史策略不受影响。
- 依赖旧 Web/引擎缺字段兜底的历史回测，与修复后的默认回测不可直接横向比较。
- CHANGELOG 必须明确标记“默认口径收口”，防止用户把数字变化误认为策略优化收益。

## 4. 选定方案

采用“**权威 YAML + 最小代码兜底对齐 + 契约测试**”：

1. `config/default.yaml` 继续保持 0.035 / 0.01，不再改动。
2. 引擎和摘要函数的本地防御性兜底改为同值。
3. 后端请求模型把未提供字段或显式 `null`（Pydantic 中均为 `None`）统一视为“使用默认值”。
4. HTML 页面首次打开、没有 localStorage 时，初始值改为 3.5 / 1。
5. 不引入新的配置归一化框架，不扩大 P0 回归面。
6. 不迁移、覆盖或删除用户保存的旧值。

## 6. 数据流

### 6.1 默认配置路径

`config/default.yaml (0.035/0.01)` → `ConfigLoader` 深度合并 → `Pipeline.step2_backtest()` → `BacktestEngine.run()` → 核心循环。

### 6.2 Web 未提供或传 null

请求没有字段或传 `null` → Pydantic 字段为 `None` → `StrategyConfig.get()` 返回 0.035 / 0.01 → `_config_to_yaml_dict()` 生成完整 stop_loss → Pipeline 深度合并 → 引擎收到 0.035 / 0.01。

### 6.3 直接引擎缺字段路径

调用者传 `{"trailing_stop": {"enabled": true}}` → `run()` / `run_cached()` 的防御性兜底返回 0.035 / 0.01。

### 6.4 显式配置路径

调用者明确传 0.08 / 0.05、0.039 / 0.017 或任何其他非 `None` 值 → 所有层原样透传，不被默认值覆盖。

## 7. 向后兼容硬约束

1. `BacktestEngine.run_cached()` 的 9 个旧位置参数不变。
2. 新能力 keyword-only 参数不变。
3. `PipelineResult`、`BacktestResult` 和 API 响应字段不变。
4. 回测仍以信号日 T 收盘价买入。
5. 已保存的策略 YAML、`config/current.yaml` 和 localStorage 值不迁移、不覆盖。
6. localStorage key 与字段 ID 不变；老用户已保存的 8% / 5% 仍按显式配置执行。
7. 老用户若要采用新默认，需要主动点击“恢复默认配置”；CHANGELOG 明确说明，不做无法区分“旧默认”与“用户主动选择”的强制迁移。
8. 显式 8% / 5% 配置改前改后核心参数和合成回测结果必须一致；仅缺字段行为改变。
9. `_simulate_core_v3`、`_simulate_core_v3_legacy` 的签名和实现不改；不新增生产直调私有 legacy 入口。
10. 不在本 P0 中顺手修共享常量、JSON 序列化、假测试等其他根因。

## 8. 错误处理

- 不新增宽泛 try/except。
- 缺字段或 `None` 使用明确默认。
- 字段存在但类型错误时保留现有 fail-fast 行为，不静默归一化非法值。
- `trailing_stop` 整体结构损坏时不在本轮补容错，避免掩盖错误配置；独立输入校验另立计划。
- 前端输入继续按现有百分数到小数的转换处理。
