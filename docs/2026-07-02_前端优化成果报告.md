# VERA 前端优化 — 业务开展及成果报告

> **项目**：VERA 量化回测系统
> **版本**：v3.3 → v3.4（前端重构版）
> **日期**：2026-07-02（周四）
> **范围**：`web/index.html` → `web/index.html` + `web/vera-ui.js`
> **耗时**：跨 3 轮对话，总墙钟约 2.5 小时（含审计、修订、实施）

---

## 一、项目背景

VERA 是一套个人实盘量化回测系统，前端为单文件 HTML（877 行，CSS + JS 全内联），通过 ECharts 可视化回测结果，FastAPI 后端提供 API。系统核心用户为上市公司董秘，日常使用场景为桌面 Chrome 浏览器。

在长期迭代中，前端累积了以下问题：

- **数据计算错误**：月度收益使用简单加总而非复合收益，长期回测偏差可达 0.5%+
- **交互缺失**：配置编辑无法取消、无恢复默认、错误提示用阻塞式 alert
- **图表粗糙**：无缩放、颜色不跟主题、基准线断裂
- **视觉扁平**：10 个 KPI 等宽排列无主次、主题按钮藏得深、切换主题后刷新丢失
- **可用性差**：交易表无筛选、退出原因文字过长截断、日志不自动滚动

用户提出"深度研究前端界面"的需求，要求找出布局、按钮、图表等方面的优化点。

### 3.2 CRITICAL 修正详情

#### C1 — 基准线二次归一化错误

**原方案**：找到基准首个有效数据后，用 `v - bmStart` 归一化（强制从 0% 起步）

**审计发现**：后端 `server.py:379` 返回的 `index_close` 已经是归一化值，前端 `(r.index_close - 1) * 100` 已转为百分比偏离。再减 `bmStart` 会把科创 50 在策略开始前的涨幅全部抹掉。

**类比**：相当于把上证指数在 2020-2024 年的涨幅归零，从策略开始日重新算起——与上证放在同一图里完全不可比。

**修正**：只裁掉开头无数据的 null 段，`connectNulls: true` 跳空连线，不做 `v - bmStart`

```javascript
// 修正前（错误）
const bmPct = bmAligned.map(v => v !== null ? v - bmStart : null);

// 修正后（正确）
const bmData = new Array(firstValidIdx).fill(null).concat(bmVals.slice(firstValidIdx));
// connectNulls: true
```

**影响**：如果照原方案执行，所有含基准对比的权益曲线图都会产生视觉错误。

#### C2 — 恢复默认值前端硬编码 vs 后端三处打架

**原方案**：前端硬编码 30 个默认值（如 `cfgCondTimeProfit: '2'`）

**审计发现**：后端存在三处默认值互不一致：

| 位置 | 字段 | 值 |
|------|------|----|
| 前端 HTML | `cfgCondTimeProfit` | `2`（代表 2%） |
| 后端 DEFAULTS (`server.py:82`) | `cond_time_profit` | `0.02`（= 2%） |
| 后端 `get()` fallback (`server.py:152`) | `cond_time_profit` | `0.01`（= 1%） |

前端写 2%，后端 `get()` 实际执行 1%——差一倍。

**修正**：调后端已有的 `/api/config/defaults` 端点（`server.py:159`），不硬编码。映射逻辑在后端 YAML 结构 → 前端字段之间做转换。

#### C3 — 主题切换无持久化 + FOUC

**原方案**：`:root` 改为 light 色值，`[data-theme="dark"]` 覆盖暗色

**审计发现**：
1. `toggleTheme()` 只改 DOM 不写 localStorage，刷新后主题丢失
2. 换 `:root` 为 light 后，暗色用户每次刷新先闪白再手动切暗色

**修正**（两步）：
1. `toggleTheme` 加 `localStorage.setItem('vera_theme', next)`
2. `<head>` CSS 之前加内联 script 读取 localStorage 设置 `data-theme`

### 3.3 HIGH 修正摘要

| 编号 | 问题 | 修正 |
|------|------|------|
| H2 | 取消按钮二次编辑覆盖快照 | 加 `if (!el.dataset.original)` guard + 编辑时隐藏编辑按钮 |
| H3 | toast 只替换 alert 自己又加 confirm | 明确：alert → toast，confirm 保留（阻塞是二次确认的预期行为） |
| H4 | KPI 分级布局缺 HTML diff | 补完整 HTML 前后对比 |
| H5 | hex+alpha 后缀 ECharts 不兼容 | 封装 `hexToRgba()` 辅助函数 |

### 3.4 审计最核心论点

> "在后端 bfill 前视偏差和成交价乐观偏差未修复之前，P0 的数据准确性修复是伪命题——用精度掩盖准确度。"

**类比**：后端给你一把偏了 10° 的尺子，前端现在量得更精细了（毫米→微米），但角度偏了 10° 没变。

**处理**：P0 照做（复合公式本身就是数学错误，简单加总本身就是错的），但在计划书和本报告中明确声明：

> ⚠️ 数据准确性仅限前端计算逻辑，后端数据源偏差（bfill 前视、乐观成交）见全局架构审计报告。

- 首次编辑时快照字段值到 `data-original` 属性

- 调 `fetch('/api/config/defaults')` 获取后端默认配置

- 清除 localStorage 中的用户自定义配置
- 失败时用 toast 提示，不阻塞

**P1-5 校验规则**：

| 字段 | 规则 | 错误提示 |
|------|------|----------|
| 起始/结束日期 | `/^\d{8}$/` | "日期格式：8位数字 YYYYMMDD" |

**P2-2 技术细节**：

原方案用 `c.down+'99'`（8 位 hex），审计指出 ECharts SVG 模式不支持。修正为 `hexToRgba(c.down, 0.6)` 格式：

```javascript
function hexToRgba(hex, alpha) {
  const r = parseInt(hex.slice(1,3), 16);
  const g = parseInt(hex.slice(3,5), 16);
  const b = parseInt(hex.slice(5,7), 16);
  return 'rgba(' + r + ',' + g + ',' + b + ',' + alpha + ')';
}
```

盈亏分布渐变色：
```
亏损区间（从深到浅）：c.down(1.0) → hexToRgba(c.down, 0.7) → hexToRgba(c.down, 0.5)
盈利区间（从深到浅）：c.up(1.0) → hexToRgba(c.up, 0.8) → hexToRgba(c.up, 0.5) → hexToRgba(c.up, 0.35)
```

退出原因饼图颜色：`[c.up, c.warn, c.down, c.accent2, c.accent, c.text2]`（6 色从 CSS 变量动态取值）

**P3-3 FOUC 修复细节**：

```html
<!-- 在 CSS 解析前设置主题，消除白屏闪烁 -->
<script>
  (function() {
    var t = localStorage.getItem('vera_theme') || 'light';
    document.documentElement.setAttribute('data-theme', t);
  })();
</script>
```

时序：
1. HTML 解析 → 执行内联 script → 设置 `data-theme` → 无闪烁
2. CSS 解析 → `:root`（light）或 `[data-theme="dark"]`（dark）立即生效
3. 字体加载（非阻塞）→ 不影响渲染

## 五、排除项与推迟项

| 项目 | 决定 | 理由 |
|------|------|------|
| 移动端响应式 | 不做 | VERA 是实盘工具，用户只用桌面 |
| 完整文件拆分（CSS/HTML/JS 全拆） | 仅拆 JS | CSS 和 HTML 合在一起 441 行可控，全拆增加构建复杂度 |
| Firefox 滚动条适配 | 不做 | 用户 95%+ 用 Chrome/Edge |
