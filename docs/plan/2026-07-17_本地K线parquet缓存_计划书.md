# 本地 K 线 Parquet 缓存 — 计划书

> 版本 v1 · 2026-07-17 · 作者 VERA
> 目标：把 TDX K 线落盘本地 parquet，回测优先读本地、miss-fetch 增量补 TDX，消除"运行时 TDX 临时缺数据导致 002008 顺延 bug"这一类问题，并大幅降低回测墙钟（数据拉取是当前瓶颈）。

## 1. 背景与目标

### 1.1 现状痛点
1. **每次回测重拉 TDX**：`DataFetcher.get_kline` 每次实时调 TDX，K 线不缓存（`data_fetcher.py` 仅 DataCache 缓存板块/简称元数据）。批量回测重复拉同一批数据。
2. **002008 类 bug 的根因**：运行时 TDX 5m 数据 6.23–6.29 缺口（盘后数据未下载全），`_build_entry_signals` 把 6.23 信号顺延到 6.30。已修（commit 021aa6c 丢弃+告警），但**根因（数据缺失）未治**——下次 TDX 缺数据还会丢信号。
3. **5m 全量拉取卡死**：`get_kline_windowed` 只拉信号后 45 天窗口，是绕开"4889 股 × 4.5 年 5m 全量 ~15 亿点拉取卡死"的 workaround，引入了窗口尾部强平失真。
4. **回测墙钟瓶颈在数据拉取**，不在循环（VERA 性能审计：循环 1.6x 退化但绝对值快，TDX 拉取才是大头）。

### 1.2 目标
- **G1 数据复用**：K 线一次下载、本地永久复用，回测不再每次重拉 TDX。
- **G2 数据完整性可检可补**：缺日检测 + 自动补拉 + 告警，从数据层杜绝 002008 类顺延 bug。
- **G3 向后兼容**：`get_kline` 签名不变，调用方零改动（实测盘点：生产 4 处 + 测试 4 处，见 §7.4；铁律：不破坏现有功能）。
- **G4 顺带可优化**：缓存就位后可考虑 numba 加速循环（非本计划范围，后续单项）。

### 1.3 非目标
- 不改回测核心循环（`_simulate_core_v3` / `BacktestLoop`）。
- 不改复权口径（仍统一前复权）。
- 不做 numba（另立单项）。
- 不做选股端改造（选股走 formula API，另议）。

## 2. 决策摘要（已与用户确认）

| 决策点 | 选择 | 影响 |
|---|---|---|
| 存储范围 | **全 A 全周期**（~5000 股 × 1d + 5m 全历史） | 5m 全量约 5–10GB parquet，一劳永逸，可甩掉 windowed workaround |
| 复权口径 | **只存前复权**（与 engine 硬编码 `front` 一致） | 简单；新分红送股时历史前复权价会变，需 staleness 检测 + 重拉 |
| 更新机制 | **运行时按需 miss-fetch 增量** | 用户无感、无需新流程；首次跑慢（全量回填），后续命中缓存 |

## 3. 存储设计

### 3.1 目录结构
```
data/kline_cache/
├── manifest.db                   # 全局清单 sqlite（WAL）：stock_code, period, last_date, last_close, fetched_at, rows, intact
├── calendar/
│   └── trading_days.parquet      # 交易日历（date, is_open）
├── 1d/
│   ├── 002008.SZ.parquet
│   ├── 600519.SH.parquet
│   └── ...
└── 5m/
    ├── 002008.SZ.parquet
    └── ...
```

### 3.2 单股 parquet schema
列：`date`(datetime64[ns], 00:00:00 归一化), `open`, `high`, `low`, `close`, `volume`, `amount`（全 float64，前复权）。
- 1d：`date` 为当日 00:00:00。
- 5m：`date` 为 bar 起始时间（9:35, 9:40, …, 15:00），与现行 engine `fill_data=False` 拉到的 index 口径一致。
- 排序：按 `date` 升序，去重（同 date 取最新）。
- 压缩：`snappy`（parquet 默认，读取快）。

### 3.3 manifest（全局清单，sqlite）
per (stock_code, period) 一行：
- `last_date`：已缓存的最晚 bar 日期
- `last_close`：该 bar 的前复权收盘（staleness 检测用）
- `first_date`：最早 bar 日期
- `rows`：行数
- `fetched_at`：最后一次写入时间戳
- `intact`：完整性 flag（true=经完整性校验无缺日/缺 bar）

manifest 用 sqlite（`manifest.db`，WAL 模式），支持并发读 + 串行写事务（见 §4.2），Python 标准库自带无新依赖。

### 3.4 交易日历
- 来源：TDX `get_trading_dates`（`core/data_fetcher.py:526`），落盘 `calendar/trading_days.parquet`。
- 每次启动/每日首次访问时增量更新到最新。
- gap 检测的基准：缓存数据的日期集合 ⊆ 交易日历 ∩ [first_date, last_date]，缺日即报警 + 补拉。

## 4. 数据流

### 4.1 读路径（`get_kline` 改造后）
> 伪代码简写；实际签名见 `core/data_fetcher.py:55`（`start_time`/`end_time`/`count`/`fill_data`/`field_list` 等参数保持不变）。

```
get_kline(stock_list, start, end, period, dividend_type="front")
  ├─ 1. 查 manifest → 分桶：cached / stale / missing
  ├─ 2. cached 股：直接读 parquet [start, end] 切片
  ├─ 3. missing 股：TDX 拉全历史 → 写 parquet + 更新 manifest
  ├─ 4. stale 股（last_date < end 或 staleness 触发）：TDX 增量拉 [last_date+1, end]
  │     ├─ 4a. 比对重叠 bar（last_date）的 close：一致 → append；不一致 → 分红shift → 全量重拉
  │     └─ 4b. gap 检测：增量段日期 vs 交易日历，缺日告警 + 补拉
  ├─ 5. 合并所有股 → 对齐到请求的 (date_index, stock_cols) → 返回 dict[DataFrame]（原签名）
  └─ 6. 异步（可选）：manifest 落盘
```

### 4.2 写路径（原子写 + per-stock 串行化）
- 单股写入：`tmp = code.parquet.tmp` → `pyarrow.write_table` → `os.replace(tmp, code.parquet)`（原子替换，读侧永不读到半成品）。
- **per-stock 串行化**：同股并发写可能因 start/end 不同拉到不同时间段，后写者覆盖前写者导致长数据被短数据回退（last-writer-wins 不安全）。方案：per-stock 文件锁（`fcntl.flock` / Windows `msvcrt.locking`），同股同一时刻只有一个 writer；不同股无锁并行。
- **manifest 升级为 sqlite + WAL**（替代原 parquet manifest）：manifest 是全局单文件，多 subprocess 并发读-改-写会丢更新（temp+rename 只保证单次写原子，不保证读-改-写事务隔离）。sqlite WAL 模式天然支持并发读 + 串行写事务，且 Python 标准库自带无需新依赖。原 §3.3 manifest.parquet 改为 `manifest.db`。
- 读侧无锁：parquet 快照读 + sqlite WAL 读，容忍并发 writer。

### 4.3 miss-fetch 策略
- **per-stock 粒度**：一只股一次 TDX 调用拉全历史（1d 或 5m），写入。下次同股直接读盘。
- **批量拉取**：missing 股列表按 500 只一批调 TDX（TDX 单次上限），减少调用次数。
- **失败重试**：单股 TDX 失败重试 2 次，仍失败则跳过 + manifest 标 `intact=false` + 告警，不阻塞其他股。

## 5. 复权处理（前复权 staleness）

### 5.1 问题
前复权价依赖"未来"的分红送股事件。新分红发生后，历史前复权价整体平移。若只增量 append 新日期，缓存的旧历史仍是旧复权因子 → 价格不连续（跳空）。

### 5.2 staleness 检测（轻量，⚠️ Phase 3 前置实测）
- 增量拉取时，TDX 返回的 `[last_date+1, end]` 段**假设**含 `last_date` 当日数据（TDX 前复权重算过）。
- 比对：`cached_last_close` vs `tdx_last_close`。
  - 一致 → 无新分红 → 直接 append 增量段。
  - 不一致 → 新分红 shift → **该股全量重拉**（删旧 parquet → TDX 拉全历史 → 重写）。
- 成本：增量段本身就含重叠 bar，零额外 TDX 调用。
- **⚠️ Phase 3 启动前必做实测**：用已知分红股（如 600519）分别以 `start=last_date` 和 `start=last_date+1` 调 TDX `get_market_data`，确认返回段是否含重叠 bar、close 是否一致。若 TDX 不返回重叠 bar，则 staleness 改为查 TDX 分红事件表（`get_divid_factors`，`data_fetcher.py:562`）判定。实测结果写回本节再开工。

### 5.3 全量重拉触发条件
1. staleness 检测不一致（§5.2）。
2. `force_refresh=True`（手动）。
3. manifest `intact=false`（之前拉取失败/缺日）。

## 6. 数据完整性 — 治 002008 类 bug（核心）

### 6.1 gap 检测（1d 缺日 + 5m 缺 bar 两套逻辑）

**1d 缺日检测**：用交易日历校验 `[first_date, last_date]` 内的日期集合，缺日即报警。

**5m 缺 bar 检测**（002008 直接受益）：5m 的完整性不是"缺日"而是"缺 bar"——每个交易日应有 48 根 5m bar（9:35–15:00）。002008 那种"日期都在、但 6.23–6.29 整周 0 根 bar"的场景，纯缺日检测检不出。逻辑：
1. 按 `date.normalize()` 分组，统计每个交易日的 5m bar 数。
2. 预期 bar 数：正常交易日 48 根；停牌日 0 根（用 `tradable` 判定，不算缺）；半天交易日（如节前）按实际交易时段算。
3. 实际 bar 数 < 预期 → `logger.warning("kline_gap_5m: %s %s 缺 bar，日期 %s 预期 %d 根实际 %d 根", ...)`。
4. 自动补拉缺 bar 段（TDX 指定 start/end 重拉，覆盖写入）。
5. 补拉仍缺 → manifest `intact=false` + 显眼告警。

**统一告警**：1d 缺日 / 5m 缺 bar 都走 `kline_gap` 前缀日志 + manifest `intact=false` 兜底。

### 6.2 与 engine 的衔接
- `_build_entry_signals` 的"信号日缺失丢弃+告警"（commit 021aa6c）**保留作兜底**。
- 缓存层 gap 检测是**第一道防线**——数据进 engine 前就补全/告警，engine 层的丢弃理论上不再触发。两层防御。
- 运行时若 engine 仍触发 `entry_signal_drop`，说明缓存层漏网（如 TDX 源头就没数据），日志双向可追溯。

### 6.3 完整性报告
- 每次回测后，`KlineCache.report()` 输出：本次命中缓存数 / miss-fetch 数 / 重拉数 / 缺日数 / intact=false 股数。
- 落盘到 `output/logs/kline_cache_report_{timestamp}.json`。

## 8. 并发与批量

### 8.1 subprocess 批量场景
- VERA 批量靠 subprocess 调 `tools/gs_run_one.py`（CLAUDE.md 明示 `Pipeline.run()` 不适合高频复用）。
- 多 subprocess 并发写同股 parquet：§4.2 per-stock 文件锁串行化同股写，不同股并行；manifest 走 sqlite WAL 事务，不丢更新。
- **预热建议**：批量前先跑一次预热（见 §9），批量阶段只读不写，零冲突、零锁等待。

### 8.2 进程内并发
- `KlineCache` 读路径无锁（parquet 快照读）；写路径原子替换。
- 不引入 threading lock（避免死锁，符合 bug-fix-workflow 线程注意事项）。

## 9. 首次迁移与预热

### 9.1 首次全量回填

## 13. 风险与回滚

| 风险 | 概率 | 缓解 |
|---|---|---|
| 前复权 staleness 漏检 → 历史价不连续 | 中 | §5.2 重叠 bar 比对 + Phase 3 前置实测 + parity 测试守卫；实测不过则改查分红事件表 |
| 并发写冲突 / manifest 丢更新 | 中 | §4.2 per-stock 文件锁 + sqlite WAL 事务；批量前 warmup 只读 |
| 5m 缺 bar 漏检（002008 类） | 中 | §6.1 5m 按 bar 数检测（不只缺日）+ 自动补拉 + intact 标记 |
| TDX 5m 全量拉取慢/卡 | 中 | 可恢复预热 + per-stock 断点续传 |
| parquet 读取与原 pandas 行为细微差异（dtype/索引） | 中 | test_return_signature_unchanged + parity 测试 |
| 缓存膨胀超盘 | 低 | 4–7GB 可接受；后续可加保留窗口清理 |

**回滚**：`use_cache=False` 一键回退原 TDX 直拉路径；parquet 文件删除即回原状。零代码回滚成本。

## 14. 明示假设

1. **TDX 仍是唯一数据源**：缓存未命中时回退 TDX；不做多数据源。
2. **股票池 = 全 A + ETF + 可转债**：全 A 取 TDX 板块遍历（`get_sector_list` + `get_sector_stocks`，`data_fetcher.py:418/437`）去重，或 `get_stock_universe` 的全市场 list_type；ETF/可转债单独取其板块（CLAUDE.md 关注 A 股/ETF/可转债，一并缓存）。港股/美股不在范围。实际取池 API 在 Phase 4 预热工具落地时确认。
3. **前复权口径不变**：engine 永远 `dividend_type="front"`；若未来要支持不复权/后复权，需升级为"原始价+因子"方案（本计划非目标）。
4. **5m 历史深度 = 3 年**：TDX 5m 数据天然有限，3 年足够策略回测；更早的 5m 取不到视为正常。
5. **首次预热可接受数小时**：用户已知"全A全周期"代价。
6. **manifest 用 sqlite（WAL）**：Python 标准库自带、支持并发读+串行写事务；per-stock parquet 文件锁保证同股写串行化。无新依赖。
7. **不改 engine 核心**：缓存只改数据获取层，`_build_entry_signals` 的丢弃兜底保留。
