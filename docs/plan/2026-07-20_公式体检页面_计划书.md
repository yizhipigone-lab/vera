把 `python tools/formula_lab.py` 的能力包进 web:新页签提交公式 → 后端队列严格串行执行(缺基线自动先补)→ 前端实时进度 → 完成后规则 JSON 自动供回测页过滤区使用。**页面上零命令行完成全流程。**

## 2. 设计

### 2.1 后端(server.py + 新模块 `tools/lab_runner.py`)

**任务队列(严格串行)**:
- 单例 `LabQueue`:内存队列 + 工作线程,**同一时间只跑一个任务**。
- **互斥(v2 修正,不对称规则)**:
  - 体检提交 → 永远 FIFO 排队(回测在跑也排,状态显示"回测运行中,体检排队中");
  - 回测提交(`POST /api/run`)在体检运行中 → **409**(与现有并发行为一致:[server.py:330-331] 已是 `pipeline_status.running → 409` 模式,该 guard 扩展检查 `lab_status.running`);
  - 体检 worker 启动任务前等待 `pipeline_status.running == False`。
- 任务生命周期:`queued → baseline(如需) → lab → done/failed`;失败不阻塞后续任务。
- **baseline 自动串联**:提交时检查 `data/baseline/{formula}_selections_{tag}.parquet` 两窗口是否都在;缺则先跑 `run_upn_baseline --formula X --start --end`(--formula 模式见 [tools/run_upn_baseline.py:46-47]),再跑 `formula_lab`。
  - **⚠️ universe 陷阱(v2 新增)**:`--formula` 模式的股票池 = `config/default.yaml` 的 universe(该文件当前被本地改为 `type: "23"` 沪深300)。队列卡片显示"基线股票池: {universe.type}",页面提示"如需全A(type=50)请先改回 default.yaml"。
- **进度解析**:formula_lab stdout 的 `[S0]/[S2]/[S3]/[S4]/[S5]` 标记([tools/formula_lab.py:264-320] 已验证)+ S4 子进程逐臂完成行 → 阶段级进度(S2 30%, S4 60%, 其余 10%);解析失败降级为"运行中(阶段未知)",不中断任务。
- **公式名校验(v2 新增)**:入队前校验 `^[\w一-龥\-]+$`(规则 JSON/报告均按公式名拼路径,防路径穿越)。

**端点**:
| 方法 | 路径 | 说明 |
|---|---|---|
| POST | `/api/lab/run` | body: `{formulas: ["A","B"], tag, tag2}` → 入队,返回队列位置;窗口缺省=近1年/近3年(按当日滚动) |
| GET | `/api/lab/status` | 当前任务(公式/阶段/已用时/预计剩余) + 队列快照 |
| GET | `/api/lab/history` | 历史体检列表:扫 `output/reports/*_filter_rules.json` + `docs/audit/*因子体检报告.md`,按公式聚合(最近时间/规则数/通过数) |
| GET | `/api/lab/report?formula=X` | 返回该公式最近一次体检报告 markdown(前端渲染) |

### 2.2 前端(独立页签)

`web/index.html` 顶部加页签栏:`[回测] [公式体检]`;体检页结构:

```
┌─ 公式体检 ─────────────────────────────────────────────┐
│ 公式名(逗号分隔多个): [________________]  [▶ 开始体检] │
│ 窗口: 短窗[近1年▾] 长窗[近3年▾](默认自动,可手填)        │
├─ 队列 ─────────────────────────────────────────────────┤
│ ▶ #2 UPN     S4 终审中(已 18 分钟)                     │
│ ✓ #1 QUANTQQ 完成 41 分钟 → 4 规则 3 通过 [查看报告]   │
│ ✗ #0 OLDFX   S0 拒跑: 信号量不足(87<200)               │
├─ 历史体检 ──────────────────────────────────────────────┤
│ QUANTQQ  2026-07-20  4 规则/3 通过  [查看] [重新体检]   │
└─────────────────────────────────────────────────────────┘
```

- 轮询 `/api/lab/status`(2s);页面可关闭,任务在 server 端继续;
- "查看报告"渲染 markdown(纯前端简单渲染或 `<pre>` 原样,不做重型 md 渲染器);
- 完成后 toast 提示"规则已登记,回测页可用"。

### 2.3 边界与失败语义

- S0 拒跑(信号少/基线生成失败):任务标 failed,原因入队列显示,不影响后续任务;
- TDX 不在线:baseline 串联失败即 failed,提示"需通达信客户端在线";
- server 重启:内存队列丢失(接受,任务本来就要人在场盯);不做持久化队列(过度工程);
- 同一公式重复提交:去重(队列中已有同公式 pending 任务则拒绝)。

## 4. 风险控制

| 风险 | 控制 |
|---|---|
| 长任务阻塞其他请求 | 工作线程 + 严格单任务;端点全异步,不阻塞 FastAPI 事件循环(subprocess 放线程池) |
| 与 /api/run 互踩 TDX/CPU | 全局互斥锁,提交时 409 明确提示 |
| 进度解析脆(formula_lab 输出格式变) | 解析失败时降级为"运行中(阶段未知)",不中断任务 |
| 并发改同一 rules JSON | 串行队列天然防;report-only 模式不在此页面开放 |

## 5. 明确不做

- markdown 富渲染(表格美化)——第一期 `<pre>` 原样;
- 并行多任务(用户已拍板严格排队);
- 移动端适配。
