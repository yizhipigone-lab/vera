# UPN + 三因子评分相关性分析 计划书

> **状态**:待审(2026-07-19)
> **关联**:[腿B数据源验证报告](../audit/2026-07-19_腿B数据源验证报告.md)(10/10 接口可用)、[三因子综合评分冲烟](../../tools/smoke_event_factors.py)(评分模型已验证)
> **目标**:验证三因子评分与 UPN 选股收益的相关性,用数据决定后续评分怎么用
> **遵循**:plan-audit-iteration 规则(写→审→迭代→交付);不破坏现有功能(零侵入回测)

---

## 0. 一句话目标

跑一版 UPN 基线回测(1 年),给每个选股信号算三因子评分,看**"评分高的信号是否收益更高"**。**先用数据验证有效性,再决定过滤/排名怎么用**——避免盲目改造策略。

---

## 1. 为什么"先看相关性"(用户决策)

不直接做"加评分过滤 → 对比 fitness",而是先做相关性分析。理由:

| 维度 | 直接过滤对比 | 先看相关性(本方案) |
|---|---|---|
| 回测逻辑改动 | 要改 selections(加阈值) | **零改动**(只读结果) |
| 调参负担 | 阈值/N 值要扫 | 无(只算相关) |
| 风险 | 改了可能无效白改 | 零(只分析) |
| 结论可信度 | 单点 fitness 对比 | 统计相关性 + 分组对比 |

**先证明"评分和收益有关",再投入过滤/排名的调参**——这是更稳的工程路径。

---

## 2. 整体设计

### 流程

```
UPN 公式 → StockSelector.run() → selections(信号 + 信号日)
                                        ↓
              BacktestEngine.run() → 每笔交易收益
                                        ↓
   对每个信号算三因子评分(T-1 数据,前视防护)← tushare 历史数据
                                        ↓
   分析:评分分组 vs 收益(高分组收益是否更高)+ 相关系数
```

### 三因子评分(复用冲烟已验证的模型)

| 因子 | 字段 | 评分规则 |
|---|---|---|
| 资金流 mf_score | net_mf_amount | +2 / +1 / -1 / -2(按净额分档) |
| 龙虎榜 dragon_score | 机构席位汇总 net_buy | +1(净买)/ -1(净卖)/ 0 |
| 大宗 block_score | 买方机构 − 卖方机构 | +1 / -1 / 0 |
| **total** | — | 三者之和,范围 [-4, +4] |

### 前视防护(铁律)

- 信号日 T 的评分**只用 trade_date < T 的数据**(T-1 及更早)
- 这些因子都是 T 日盘后数据,T 日收盘选股时还没出 → 必须 T-1
- 与冲烟验证一致([tools/smoke_event_factors.py](../../tools/smoke_event_factors.py) 的 `trade_date < SD` 逻辑)

## 3. 实施步骤

### 阶段 A:历史数据准备

- **区间**:近 1 年(精确起止实施时确认 tushare 数据范围,预计 2025-07 ~ 2026-07)
- **三因子**:龙虎榜 top_list/top_inst、大宗 block_trade、资金流 moneyflow
- **拉取策略(审计 H3 修正)**:按 `trade_date` **逐日拉**(非 ts_code 分批)
  - 审计发现:三因子 endpoint 单日数据量 5090/650/133 **均远低于 6000 上限**("6000 坑"是 share_float 独有,与三因子无关,之前计划书写错了)
  - 按 trade_date 逐日拉 1 年 ≈ 244 次 API 调用,比 ts_code 分批(5090 次)少 **20 倍**
  - 加 sleep 避免频控(tushare 每分钟调用限制)
- **前置验证(审计 M2)**:先拉 1 周确认 tushare 历史深度够 1 年,再全拉
- **缓存**:`data/factors/{mf,dragon,block}_<yyyy>.parquet`
- **脚本**:`tools/fetch_event_factors.py`(新)

### 阶段 B:UPN 基线回测(新脚本拿 per-trade 明细)

- **新写 `tools/run_upn_baseline.py`** 直接调 `BacktestEngine.run`,落盘 selections + trades_df
- **审计 H1**:`gs_run_one.py` stdout 只输出聚合 JSON(cumret/annret/...),**无 per-trade 明细、无 selections 落盘**;per-trade 真实来源是 engine.py `_build_trades` 返回的 trades_df(stock_code / entry_date / exit_date / profit_pct)
- `gs_run_one.py` **不动**,只作"StockSelector → selections → BacktestEngine.run"调用模式参考
- 基线聚合指标(年化/回撤/夏普/Calmar/胜率)同时输出 — 满足 CLAUDE.md 铁律 4(审计 M3)

### 阶段 C:评分 + 相关性分析

**信号→交易匹配规则(审计 H2)**:

- trades_df 有 `entry_date` **没有 select_date**,匹配规则:按 `(stock_code, entry_date ≤ T+3)` 匹配(T=信号日)
- 多笔交易取首笔;0 笔计 NaN 并单报"信号成交率"(涨停/停牌致未成交)
- 4 种边界全覆盖:0 笔 / 1 笔 / N 笔(止盈后再买)/ 多信号同一笔

**评分**:

- 给每笔信号算 `total_score`(信号日 T,用 T-1 数据)
- **评分模块从冲烟抽取时重构(审计 M1)**:冲烟用全局 SD 常量,生产改为 **per-row select_date**(每个信号用自己的信号日判断 `trade_date < T`),避免前视 bug

**分析**:

- **分组对比**:按 total_score 分 3 组
  - 高分组(total ≥ 2)
  - 中分组(-1 ≤ total ≤ 1)
  - 低分组(total ≤ -2)
  - 对比每组的:平均收益 / 胜率 / 夏普 / 样本量
- **相关系数**:total_score vs 单笔收益(Spearman 秩相关,适合有序分数)
- **显著性**:高分组 vs 低分组收益差异,**p<0.05 且均值差≥3% 双条件**才算显著有效(审计 M4)

### 阶段 D:决策建议(交付物)

据相关性结果给结论:

| 结果 | 后续动作 |
|---|---|
| 正相关显著(高分组收益明显高于低分组) | 评分有效 → 下个计划书做过滤/排名 |
| 无相关 / 负相关 | 评分无效 → 换因子或调评分逻辑(如调权重/分档) |
| 样本不足(低分组没几只) | 延长区间或放宽分组阈值 |

## 5. 风险控制

| 风险 | 控制 |
|---|---|
| 破坏现有回测 | 零侵入(不改 engine/selections,只读结果) |

| 不联实盘 | 纯分析,不产生交易信号 |

## 附录:统计方法说明

- **分组对比**:3 组的平均收益/胜率/夏普,直观看评分高低是否对应收益好坏
- **Spearman 秩相关**:total_score(有序)vs 收益(连续)的单调相关性,比 Pearson 更稳健(不怕离群值)
- **t 检验 + 效应量**:高分组 vs 低分组收益差异,**p<0.05 且均值差≥3% 双条件**才算显著有效(审计 M4)
- **样本量要求**:每组至少 30 笔交易才具统计意义(交易次数下限规则一致)

> 这些方法都是 code-grade(数值计算,无主观判断),符合"可测试性"要求。
