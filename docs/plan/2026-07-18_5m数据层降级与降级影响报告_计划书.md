# 5m 数据层降级 + 降级影响报告 — 工作计划书

> 版本 v1 · 2026-07-18 · 作者 VERA
> 前置共识（本次讨论确定）：每股每天粒度降级；用"数据层 1d 填充"（缺 5m 的股-日用该股当天真实 1d OHLC 填满 5m bar），不做混合网格；执行价多为派生价故近无损，唯一损失是"同日止盈止损先后顺序"（日线固有）；报告要量化这个偏差。

## 1. 背景与目标

### 1.1 背景
- TDX 5m 历史上限 2024-06-27，且存在全市场 5m 缺口周（2026-06-23~29）。
- 当前 5m 数据缺口 → `_build_entry_signals` 丢弃信号 + 告警（commit 021aa6c），会丢交易。
- 用户要：缺 5m 的股-天自动降级用日线（保住信号），其他股天照常 5m，并量化降级对结果的影响。

### 1.2 目标
- **G1 每股每天降级**：某股某天有 5m 用 5m，没有则该该股该天用 1d OHLC 填充，其他股天不受影响。
- **G2 停牌不降级**：1d 也缺（真停牌）则不填充、保持不可交易。
- **G3 降级影响报告**：降级笔数 + 影响金额上下范围 + 对盈利率等指标影响。
- **G4 opt-in**：默认关（现状丢信号），显式开启才降级。

### 1.3 非目标
- 不做全场降级、不做单信号级"半 5m 半日线"混合网格。
- 不改回测核心循环（BacktestLoop / _simulate_core_v3）逻辑。
- 不做偏差校准实验（拿完整 5m 人为降级对账）——那是一次性验证，另议。

## 2. 核心机制：数据层 1d 填充（已共识）

当某股某天 5m 缺失（5m 网格该日 48 根全 NaN）且该股当天 1d 有数据：
- 用该股当天真实 **1d OHLC** 把这 48 根 bar 填满（每根 = 当天 1d 的开/高/低/收）。
- 网格仍统一 5m（48 根/天），bpday / 时间止损 / T+1 全不乱。
- **入场** = 1d 收盘 = 当天真实收盘（符合铁律，与 5m 入场同价）。
- **出场** = 派生价（trail_line / stop_price / ladder_price）基于真实 1d 高低价，近无损。
- **标记**：`degraded_np[n_dates, n_stocks]` 布尔矩阵，标记 1d 填充的 bar。

关键正确性（已验证 + 审计修正）：
- 执行价多为派生价（trailing=trail_line、cost_stop=stop_price、ladder=ladder_price、atr=trail_line），不依赖盘中 5m 轨迹 → 降级日这些止损与真 5m 几乎一致。
- **但 time_stop / cond_time / first_day 三个策略按 bar.close 成交**（time_stop.py:27、cond_time.py:28、first_day.py:41）：真实 5m 在触发日早盘价成交，降级日 48 根 close 全=1d 收盘 → 按收盘成交。这是**系统性偏差**（方向不定），必须纳入报告（见 §3 修正），不能声称"唯一损失是同日顺序"。
- 同日"先摸高还是先破低"的顺序仍是损失之一（日线固有，与纯日线回测相同）。

## 3. 报告需求确认（三要素 + 审计修正的第四类偏差）

| 要素 | 计算方式 |
|---|---|
| **降级笔数 + 占比** | 入场 bar 或出场 bar 落在 degraded_np 上的交易数 / 总交易数 |
| **影响金额上下范围（模糊日顺序歧义）** | 对每笔降级交易，找其持有期内的"模糊日"（同日 high 够止盈档 且 low 破止损线）。模糊日上真实顺序未知：<br>· 乐观上界 = 按"先止盈"（移动止盈优先）重算该交易收益<br>· 悲观下界 = 按"先止损"（硬止损优先）重算<br>非模糊日交易收益确定（派生价），不计入区间。汇总得 [悲观总额, 乐观总额] |
| **close 价策略偏差（审计 HIGH-1）** | time_stop / cond_time / first_day 按 bar.close 成交：真实 5m 在触发日早盘价成交，降级日按 1d 收盘价成交。对这类出场单独统计"5m 早盘价 vs 1d 收盘价"的金额差，并入影响范围。5m 稀疏窗口常开时间止损（engine.py:770），是高频路径，不可漏。 |
| **对盈利率等指标影响** | 降级后实际盈利率；由上下界推出 [悲观盈利率, 乐观盈利率]；夏普/最大回撤同法给范围 |

## 4. 技术方案

### 4.1 新模块 `backtest/degrade_5m.py`（独立新文件，不动旧逻辑）

**第一步：从交易日历构建完整 5m 网格（审计 CRITICAL-1：全市场缺口日根本没行，必须先插行）。**
5m 网格的行 = TDX 实际返回 bar 时间戳的并集；全市场缺口日（6.23-6.29）在 index 里零行，无从"填"。所以先从交易日历合成完整 5m 网格：
- 对区间内每个交易日，合成 48 个 bar 时间戳（9:35, 9:40, ..., 15:00）。
- 全部矩阵 reindex 到这个完整日历网格 → 缺口日（无论全市场还是单股）都有行（NaN）。
- **不变量保护**：每天必须恰好 48 根（loop 的 T+1 `i//bpday`、首日策略 `bar_index%bpday` 都假设 48 根/天），插行位置必须正确。

**第二步：填充缺失股-天。**
```python
def apply_5m_degradation(
    close_5m, high_5m, low_5m, open_5m,   # 5m 矩阵 (DataFrame, 含 NaN, 已 reindex 到完整日历网格)
    close_1d, high_1d, low_1d, open_1d,   # 1d 矩阵 (同股票列)
) -> Tuple[close, high, low, open, degraded_np]:
    """对完整日历网格上"整日 NaN 但 1d 有数据"的股-天, 用 1d OHLC 填充 48 根 bar。
    返回填充后矩阵 + degraded_np 布尔矩阵 (n_dates, n_stocks)。
    停牌日 (1d 也缺) 不填充, 保持 NaN。"""
```
- 判定缺失：按 `close_5m.index.normalize()` 分组，某股某天 bar 数 == 0 且 1d 该天非 NaN → 该股该天降级。
- 填充：该股该天 48 根 bar 赋 1d OHLC。
- degraded_np 对应位置置 True。

**集成顺序约束（审计 MEDIUM-2）：**
- 填充必须在 engine 的 high/low `ffill`（engine.py:805-806）**之前**，否则前一日数据先 ffill 进缺口。
- 必须同时填 `close_raw`（否则 tradable_np 判降级 bar 不可交易，见 §4.3）。
- open 必须填 1d open（硬止损跳空保护 cost_stop.py:32-34 需要真实 open）；停牌日（1d 也缺）不填，保留 open=NaN 让 T+1 买入自然跳过（engine.py:807 语义）。

### 4.2 1d 数据来源
- 复用 KlineCache（全 A 1d 已预热），`DataFetcher.get_kline(period="1d", use_cache=True)` 取同股票同区间的 1d，对齐到 5m 网格的日期。
- **复权一致性断言（LOW-1）**：对 5m 和 1d 都有数据的天，比较"5m 日聚合 OHLC" vs "1d OHLC"，应精确一致（防复权口径漂移）。

### 4.3 可交易性合并（审计 CRITICAL-2，降级能真正成交的前提）
`tradable_np` 由**填充前**的 close_raw 构建，降级 bar 在里面是不可交易 → 入场被 skip（信号照样丢）、持仓可能误判退市强平（更糟）。必须照稀疏窗口 window_mask 的模式（engine.py:816-824）：
- `tradable_np |= degraded_np`（降级 bar 标记为可交易）。
- **重算 last_tradable_idx**（降级 bar 也是可交易 bar）。
- 集成点在 `_build_tradable_from_raw`（engine.py:812）之后、window_mask 合并（:816）之前。
- 这样降级日入场不再被 skip、持仓不误判退市。

### 4.4 engine 集成 + 接缝修正（审计 HIGH-2：degraded_np 不进 BacktestLoop）
- 配置开关：`degrade_5m: bool`（默认 False）+ 复用 `use_kline_cache`。
- period=="5m" 且 degrade_5m=True 时：构建完整日历网格 → 取 5m/1d kline → `apply_5m_degradation` → 得填充矩阵 + degraded_np → 按 §4.3 合并 tradable_np。
- **degraded_np 不传入 BacktestLoop**（`_simulate_core_v3` 是冻结 39 参壳，有 54 组 parity + tests/test_engine_run_path.py:45-49 签名守卫，不动签名）。
- 执行语义走 tradable_np（§4.3）；降级报告走 `_post_process`（engine.py:1125）**事后扫描** raw_trades 的 (ci, entry_idx, exit_idx)，无需 loop 内插桩。

### 4.4 降级交易追踪（事后扫描，不进 loop）
- raw_trades 每行带 (ci, entry_idx, exit_idx)（state.py:192 TradeBuffer.append）。
- 回测后在 `_post_process`（engine.py:1125）扫描：交易 (ci, entry_idx..exit_idx) 范围内 degraded_np 有 True → 标记为降级交易。
- **部分出场聚合**：阶梯部分卖产生多条同 entry_idx 的交易行（loop.py:292-304），按 (ci, entry_idx) 聚合成"持仓"再判定，不按单笔交易行。
- 统计降级笔数、占比、降级日分布。

### 4.5 影响金额上下范围计算（含审计 MEDIUM-1 重放口径）
raw_trades 只有 9 列，**没有 peak_hi / ladder_done**（trail_line、阶梯档位都路径依赖）。模糊日重算需从填充后矩阵**重放** entry→模糊日：
- **peak_hi 重放**：从 entry 到模糊日逐 bar 取 high 累计最大（trailing 的 trail_line = peak_hi×(1-drawdown)）。
- **ladder_done 重放**：逐 bar 重放阶梯触发史，判定该日 high 够哪一档。
- **口径（必须先定，直接决定下界宽度）**："先止损"分支里，**模糊日自己的 high 不计入 peak_hi**（先破低时当日 high 尚未发生，trail_line 用前一日 peak）；"先止盈"分支则计入。
- 模糊日判定：该日 `high >= 任一止盈触发价` 且 `low <= 任一止损触发价`（用重放后的真实触发价）。
- 模糊日交易：按"先止盈"/"先止损"两档重算出场（派生价），得 乐观/悲观 收益。非模糊日交易：收益确定。
- 汇总：悲观总额 / 乐观总额 / 实际总额；盈利率范围 = 总额 ÷ 初始资金；夏普/回撤按三档各算一次。

### 4.6 降级日涨停过滤（审计 MEDIUM-3）
`_filter_limit_up` 用"当根 close vs 前一根 close"判涨停（engine.py:1285-1290），降级日 48 根同价 → 日内比值恒 1.0 → **结构上不可能触发**。一只 1d 涨停的股在降级日会被按收盘价"买入"（实际买不到）。
- 方案：对**降级入场日**，额外做一条基于 1d 的涨停判定（1d close vs 前一日 1d close），命中涨停则该降级信号丢弃 + 告警。
- 集成在 §4.3 的 tradable 合并处：降级日若 1d 涨停 → degraded_np 对应位置不计入 tradable_np。

### 4.7 报告输出
- 并入现有 result：新增 `degradation` 字段（**需动 `backtest/result.py` frozen dataclass + `pipeline/result_writer.py` 序列化**，加字段向后兼容用 _UNSET 哨兵，LOW-2）：
  ```json
  {
    "enabled": true,
    "degraded_trades": 12,
    "total_trades": 334,
    "degraded_pct": 0.036,
    "impact_amount": {"pessimistic": -8230, "actual": -3100, "optimistic": -950},
    "return_range": {"pessimistic": -0.0082, "actual": -0.0031, "optimistic": -0.0009},
    "ambiguous_trades": 3,
    "close_based_stop_bias": {"count": 5, "amount_diff": -1200},
    "degraded_days": {"2026-06-23": 5, "2026-06-24": 4}
  }
  ```
  - `close_based_stop_bias`：time_stop/cond_time/first_day 三个按收盘成交的策略，降级日"5m 早盘价 vs 1d 收盘价"的金额差（审计 HIGH-1）。
- 前端报告区新增"降级影响"卡片（后续；本期先出 JSON + 日志）。

## 8. 风险与回滚

| 风险 | 缓解 |
|---|---|
| 全市场缺口日插行错位（48 根/天不变量破坏） | test_exactly_48_bars_per_day 锁死；插行从日历合成 |
| tradable_np 不更新 → 降级不生效/误判退市 | test_degraded_day_tradable + test_last_tradable_idx_recomputed 锁死 |
| 模糊日判定口径（"够得着"的阈值 + peak 口径）影响区间大小 | 用各策略真实触发价判定；"先止损分支 high 不计 peak"口径写死 |
| close 价策略降级偏差 | §3 单列统计，不掩盖 |
| 1d 与 5m 复权口径不一致 | 前复权一致 + 5m 日聚合 vs 1d OHLC 断言 |
| 降级日涨停过滤失效 | 1d 涨停判定补回 |
| 降级引入隐性乐观/悲观 | 报告给上下界，不掩盖；默认关 opt-in |

**回滚**：`degrade_5m=false`（默认）即回退现状（丢信号路径），零代码回滚成本。

## 9. 验收标准
- 降级笔数/占比/影响金额上下范围/盈利率范围/close价策略偏差 都能在 result.degradation 输出
- 全市场缺口日（6.23-6.29 那种）也能降级（插行后有行可填）——002008 场景真正修复
- 降级日入场价 = 1d 收盘；停牌日不降级；别股不受影响；降级日 1d 涨停被拒绝
- degrade_5m=False 时与现状完全一致（丢信号）；_simulate_core_v3 签名不变
- run_cached 路径不支持降级（仅 run()），文档注明（LOW-3）
- 全量 pytest 零回归

## 10. 明示假设
1. "上下范围"= 模糊日的乐观（先止盈）/悲观（先止损）两档 + close 价策略的早盘 vs 收盘差，非"整个交易日高低价全区间"。
2. 全市场缺口日通过"从交易日历合成 48 时间戳插行"获得行（CRITICAL-1 的解），每天恰好 48 根。
3. 部分 5m bar 缺失的当天按"整日缺"处理（填 1d），除非用户要求"保留已有 5m bar"（更细，后续）。
4. "先止损"分支里模糊日自己的 high 不计入 peak_hi（先破低时 high 未发生）。
5. 本期只出 JSON + 日志报告，前端卡片后续。
6. 偏差校准实验（完整 5m 人为降级对账）不在本期，另议。
