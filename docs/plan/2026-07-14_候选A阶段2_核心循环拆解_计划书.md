### 2.1 关键接口（v3 全部补齐定义）

**`TriggerResult`** — frozen dataclass，**折叠 execution_price**（v3 修正 CA4：执行价计算依赖 check 内部中间值，分离会导致数据耦合，故并入）：

```python
@dataclass(frozen=True)
class TriggerResult:
    reason: int              # 3/4/5/6/7/8/9/10/12
    strategy_name: str       # "cost_stop" / "ladder_tp" / ...（用于日志，不用于排序）
    execution_price: float   # 已算好的成交价（含跳空保护等逻辑，check 内算完填入）
    sell_ratio: float = 1.0  # 1.0=全卖；<1.0=部分卖（ladder 阶梯止盈）
    is_partial: bool = False # ladder 部分卖标记（trailing_first 双触发用）
```

**`ExitStrategy`** — Strategy 模式，**check 返回 `List[TriggerResult]`**（v3 修正 CA1：单结果 `Optional` 无法编码 trailing_first 同 bar 双触发，改多结果）：

```python
class ExitStrategy(Protocol):
    """Strategy 模式：单一止损类型的检测 + 执行价计算。
    命名上沿袭 VERA 团队习惯叫 adapter，实际为 Strategy pattern。
    """
    name: str                  # 用于日志

    def check(self, pos: Position, bar: Bar,
              ctx: Context) -> List[TriggerResult]:
        """返回 0/1/2 个触发。
        trailing_first 下 ladder 可返回 1 个部分卖触发；
        dispatcher 负责在同一 bar 内追加 trailing/cost_stop 全卖剩余的第二个触发。
        """
        ...
```

> **v3 修正要点（相对前版）：**
> - **无 `execution_price()` 方法**（CA4）——执行价在 check 内算好填进 TriggerResult
> - **无 `priority` 字段**——排序由 dispatcher 按构造时 list 顺序 + Priority 枚举编码
> - **无 `enabled` 字段**（HA1/M2）——capability gating 在 dispatcher 构造时过滤，禁用的策略根本不进 list
> - **check 返回 List 而非 Optional**（CA1）——支持双触发

**`AbsoluteStrategy`** — 单独协议，处理"先于所有 ExitStrategy 评估"的特殊情况：

```python

    VERA 当前唯一实例: formula_sell (reason=12, engine.py:24 明确):
    "formula_sell (reason=12) 始终最高优先级, 不受 priority 开关影响"
    （亦见 engine.py:46/51/174 注释）
    """
    name: str
    def check(self, pos: Position, bar: Bar,
              ctx: Context) -> Optional[TriggerResult]: ...
```

> **引用修正（v3，CA1/C1）：** 前版称"CLAUDE.md:24 明确"是错的——CLAUDE.md:24 是回测引擎架构表行，与 formula_sell 无关。规则真实出处是 **engine.py:24**（模块 docstring 原句）"，亦见 :46/:51/:174。前版审计 C1 改成"engine.py:51"也不够精确——:24 才是原句出处。本版统一引 engine.py:24。

### 4.2 行为不变的证据（parity 矩阵，v3 扩维）

阶段 4 的 parity 测试覆盖矩阵（**≥54 组**，修正前版公式错误 + 补盲区）：

```
核心正交（4 维，修正前版 ×1 bug）:
  3 priority (stop_first / ladder_tp_first / trailing_first)
  × 4 capability 开关组合 (formula_exit × gap_protection × delisting × 全 off)
  × 2 trade_count 分支 (部分卖 ladder / 全卖)
  × 2 trailing 状态 (激活 / 未激活)
  = 3 × 4 × 2 × 2 = 48 组核心对照

外加 6 组边界对照（补 risk 审计盲区）:
  + 1 多持仓 swap-and-pop 重排（5 只同时持仓，触发顺序依赖数组重排，CR2）
  + 1 连续 bar ladder 部分卖（bitmask 跨 bar 累计，bar i 部分卖 → bar i+1 再触发，CR3）
  + 1 同 bar formula_sell(12) + delisting(11) 双 pre-dispatcher（HR2）
  + 1 trailing_first 同 bar 双触发（ladder 部分卖 + trailing 全卖剩余 = 2 笔交易，CA1）
  + 1 空信号数组（entry_np 全 False，MR3）
  + 1 T+1 同日不卖（entry 与 exit 同 day 不同 sub-bar，i//bpday 约束，MR4）
  = +6 组
```

总计 **54 组 parity 对照**，全部用 `np.array_equal` 字节级断言（含 equity_curve + raw_trades 双数组）。

> **v3 修正（H4/CR2/CR3/HR2/CA1/MR3/MR4）：** 前版公式 `3×4×2×1=24` 把 trailing 状态写成 ×1 但文字又说 ×2，自相矛盾；实际 4 维正交 = 48。再加 6 组 risk 审计发现的盲区。前版"27 组"严重不足。
