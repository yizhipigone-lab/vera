## 0. 一句话结论(给时间紧的读者)

> AutoResearch 技能**不是**一个"GPU/LLM 增强平台",它只是一个**"单文件 + 单指标 + 死循环"的实验脚手架**。
> 它能 1:1 移植到 VERA 的**纯量价策略迭代**(低风险,立即可做);但**不能**直接驱动"LLM 文本因子挖掘 + 自动下发实盘"(高风险,且度量标尺对不上)。
> 推荐分两条腿:**腿 A(量价循环)先做,腿 B(文本因子)先离线建因子库、解决数据源再说**。

## 1. AutoResearch 真实能力剖析(剥掉神化)

### 1.1 它到底是什么

Karpathy 的 autoresearch,本质三件事:

| 要素 | 内容 | 备注 |
|---|---|---|
| **一个文件** | 只能改 `train.py`;评估代码 `prepare.py` 锁死 | 改动边界极清晰 |
| **一个数字** | 唯一判定标尺 `val_bpb`(语言模型预测精度),程序 `grep` 即出 | 单一标量,无歧义 |
| **一个死循环** | 改代码 → 训 5 分钟 → 看数字 → keep/discard → 无限重复 | 人睡觉,agent 跑通宵 |

### 1.2 它**不**包含(必须澄清)

- ❌ LLM 调用逻辑(没有 API 调用、没有文本生成)
- ❌ 文本爬虫 / 舆情抓取 / 多模态处理
- ❌ 因子计算 / 高维相关性训练(那是 PyTorch 训练循环本身,不是 autoresearch 的能力)
- ❌ 多目标优化(它只认单一标量)

它的全部价值 = **"自动化 keep/discard 纪律 + git 分支前进/回退 + 实验日志 + 无限循环调度"**。思想内核 borrowed 自强化学习的"棘轮循环":**只往好走,不往回退**。

### 1.3 比喻

> 你以为 AutoResearch 是一整套"AI 厨房"(买菜、切菜、炒菜全自动)。
> **其实它只是一个"自动尝味道、不行就倒掉重做"的机械臂。**
> 菜谱、食材、火候全得你自己备。它只保证"每次都比上次好,或者至少不退步"。

## 2. 用户原始方案的三处致命错配(直说)

| 错配 | 原方案描述 | 真实情况 | 后果 |
|---|---|---|---|
| **① 神化为 LLM/GPU 增强平台** | "承载 LLM 语义解析、文本因子挖掘、高维相关性训练、策略迭代优化" | 技能本身零 LLM、零爬虫、零因子逻辑 | 装了技能 ≠ 有文本因子能力,真正活儿一行没干 |

### 3.1 值得移植的真内核

剥掉宏大叙事,原方案暗合 autoresearch 的核心思想:**实验循环 + 程序化度量 + keep/discard 纪律**。量化领域叫 **walk-forward optimization / 策略迭代 harness**,真实有效。

## 4. 两条腿裁剪路线图

### 4.1 腿 A:量价策略迭代循环(autoresearch 1:1 移植)

| 维度 | 设计 |
|---|---|
| **autoresearch 原生度** | ★★★★★ 几乎 1:1 |
| **要改的"train.py"** | 公式参数 / 止盈止损 / 选股过滤条件(不动 engine 核心) |
| **判定标尺** | fitness(年化主 + 回撤硬约束),见 §6 |
| **循环单元** | 改公式/参数 → `engine.run_cached` → 看 fitness → keep(前进 commit)/ discard(reset 回退)→ 记 results.tsv |
| **数据依赖** | 已有(5m/1d K 线,K线缓存已落地) |
| **实盘风险** | 中(纯量价,口径已校准;信号日 T 收盘买入铁律保持) |
| **能否立即开工** | ✅ 阶段 1 即可 |
| **预期产出** | 1-2 周出夜间无人值守循环,跑上百轮 |

### 4.2 腿 B:LLM 文本因子(autoresearch 帮不上循环)

| 维度 | 设计 |
|---|---|
| **autoresearch 原生度** | ★☆☆☆☆ 度量仍是量价 fitness,LLM 只做离线因子表 |
| **LLM 角色** | **离线**生成因子表(日期+代码+因子值)→ 注入选股过滤,**不在线迭代** |
| **判定标尺** | 同腿 A 的 fitness,但因子先离线验证再加进循环 |
| **数据依赖** | ❌ **最大瓶颈不是算力,是文本数据源**(见 §7) |
| **实盘风险** | **高**(易过拟合 + 黑天鹅 + 主观性强) |
| **能否立即开工** | ❌ 先解决数据源 + 离线因子库 |
| **阶段** | 阶段 3+,腿 A 跑顺后 |

### 4.3 关键澄清:VERA 不走通达信原生

原方案"翻译成通达信公式丢回 CPU 回测",在 VERA 现实里**不是写 .tne 公式**,也**不是 EXTERNSTR/EXTERNVALUE**。

VERA 架构是 Python 的 [core/formula_runner.py](../../core/formula_runner.py)(TDX 公式翻译层),核心入口:
- `FormulaRunner.run_stock_selection_with_dates(...)` — 选股([core/formula_runner.py:48](../../core/formula_runner.py#L48))
- `FormulaRunner.run_indicator(...)` — 指标([core/formula_runner.py:191](../../core/formula_runner.py#L191))

文本因子的真实注入点:
1. **selections 层过滤**(推荐):在 `run_stock_selection_with_dates` 返回的选股结果上,叠加文本因子过滤(如"负面舆情标记=0 才保留")
2. **stop_config 动态止盈止损**:文本利空信号提前止损、主线热点放宽止盈
3. **不碰 TDX 公式本体**,与 `degrade_5m` 外挂数据层同一思路

## 6. fitness 函数设计(腿 A 的核心,多目标压成单标量)

### 6.1 设计原则

autoresearch 的 val_bpb 是单标量才能 keep/discard。量化的多目标必须压成一个数。推荐**主标量 + 硬约束 + 次序判定**三层结构:

```
fitness = 年化收益(主标量)
约束:最大回撤 ≤ X% (硬约束,超限直接 discard)
次序:同 fitness 时比 夏普 > 胜率 > Calmar > 交易次数(适中)
```

### 6.2 推荐默认值(可调,待用户拍)

| 参数 | 推荐默认 | 依据 |
|---|---|---|
| 回撤硬约束 X% | 15% | CLAUDE.md "中等回撤"风险偏好;gs_5m_sweep 达标线就是"年化>23%+回撤≤15%" |
| 年化达标线 | 23% | 同上,已有基准 |
| 交易次数下限 | 30 次/回测区间 | 防止"一年交易 3 次碰巧全对"的统计无意义策略。**区间最小 1 年(约 244 交易日)**,30 次下限按 1 年口径,更长区间按比例放大(如 5 年区间下限 150 次) |
| 夏普最低 | 1.0 | 机构及格线 |

### 6.3 fitness 公式(候选,待审)

**方案 A(简单加权,推荐先上)**:
```
fitness = 年化 - 0.5 × max(0, 回撤 - 15%) × 100
# 回撤超标按 0.5 倍惩罚年化;过 15% 部分每 1% 扣 0.5% 年化
```

**方案 B(夏普为主)**:
```
fitness = 夏普 × 10 - max(0, 回撤 - 15%) × 100
# 风险调整收益优先
```

**推荐先用方案 A**,理由:你的核心诉求是"年化 + 中等回撤",年化作主标量最直观。方案 B 适合后续想优化"收益稳定性"时切换。

### 6.4 牛熊分段(防过拟合,阶段 2 启用)

CLAUDE.md 明确"回测方向系统性乐观",单区间年化好不可信。阶段 2 引入:
- 拆 2018(熊)/2019-2020(牛)/2021-2022(震荡)/2023-2024(跌)/2025(修复)等区间
- **2 个以上不重叠区间 fitness 都达标才 keep**
- 自动淘汰"仅牛市有效"参数

### 6.5 异常 fitness 处理(防挂循环)

量价策略 0 信号、回测区间无交易、metrics 异常都会让 fitness=NaN 或空输出。明确处理路径:

- **fitness=NaN 或空输出 → 默认 discard**(记原因:0 信号 / 无交易 / 指标异常),**不挂循环**
- 附录 A 第 6 步"空输出=崩溃"逻辑与此一致:运行失败 → `crash`;有结果但 fitness=NaN → `discard`
- results.tsv 的 status 字段三态:`crash`(运行失败)/ `discard`(跑了但劣化或异常)/ `keep`(提升)

### 7.3 推荐起步:公告因子

**理由**:
1. **官方源稳定**:巨潮 cninfo 是证监会指定披露平台,API 免费(限频可缓存)
2. **事件类型明确**:增减持/业绩预告/ST/回购/定增,LLM 打分模板化,信噪比高
3. **董秘职业优势**:你对公告的解读能力远超散户,这是你的 alpha 来源
4. **过拟合风险低**:事件型因子比"情绪分"更难在历史回测里偶然拟合

### 7.4 LLM 处理层(符合本地优先铁律)

CLAUDE.md 保密边界:本地部署优先,策略/资金高度机密。

| 处理方式 | 适用 | 风险 |
|---|---|---|
| **本地部署**(Qwen2.5-14B / GLM-4 / DeepSeek + vLLM/Ollama) | 公告/研报文本(公开) | 文本不出本地,符合铁律 |
| API(智谱/DeepSeek/通义) | 公开新闻文本 | 文本非机密可用,**但绝不发持仓/策略信号** |

**推荐**:公告/研报(公开文本)本地部署 Qwen2.5-14B + vLLM,GPU 一次性投入,长期复用。腿 B 的"GPU 算力"主要用在这一层,而不是 autoresearch 循环。

| **margin_trade** | 融资融券 4 个 API | ★★★ | 杠杆/情绪因子 |
| **dragon_tiger_inst** | 龙虎榜机构明细 | ★★★★ | 主力动向因子 |
| finance_news | 财经新闻(百度爬虫) | ★★ | 文本原料,搜索式非全量 |
| hot_news_7x24 | 7×24 热点(1500 条/次,去重) | ★★ | 热点原料 |

**能替代商业终端的部分**(Tushare 覆盖不错):

1. ✅ **结构化事件因子**:龙虎榜、大宗交易、融资融券、资金流向、增减持(财务表股东变动)
2. ✅ **财务因子**:利润/资产/现金流结构化数据
3. ✅ **可转债数据**:有专门工具
4. ⚠️ **热点新闻原料**:有但搜索式/快照式,非全量历史归档,要 LLM 二次打分

**替代不了商业终端的部分**:

1. ❌ **研报全文**:Tushare 只有列表/评级,无全文 → 一致预期因子只能做粗版
2. ❌ **公告全文历史归档**:Tushare 有列表,全文要另接巨潮 cninfo
3. ❌ **股吧/互动易全文**:没有(本方案本就不推荐股吧)
4. ❌ **深度机构数据**(调研纪要、基金持仓明细变动):不全

**结论**:FinanceMCP 能替代商业终端的**结构化数据那一半**,替代不了**文本全文那一半**。即"能拿多少数据就多少数据"的答案 = **结构化事件/财务/资金面数据基本够用,文本全文类仍缺**。

**对腿 B 起步建议的调整(重要)**:

- **新增"结构化事件因子"作为腿 B 真正起步点**:龙虎榜/大宗/融资融券/资金流,FinanceMCP 本地部署几乎零成本即可跑起来,且**数值型直接进 selections 过滤,不依赖 LLM 打分**,比纯文本因子**更难过拟合**
- 起步顺序调整:**结构化事件因子(FinanceMCP)→ 公告文本因子(cninfo + LLM)→ 研报因子(需 Wind 或放弃)**

**Tushare 积分成本提醒**:高级接口要积分(moneyflow 2000 / moneyflow_mkt_dc 5000 / moneyflow_ind_dc 6000),学生可申请 2000 免费积分;商业用途需付费积分。这是腿 B 的真实成本,不是零。

#### 7.6.2 TrendRadar — 舆情热度层(newsnow + AI 分析)

**实质**:`sansan0/TrendRadar`(60.5k star,GPL-3.0,v6.10.0)= **AI 驱动舆情监控 + 热点筛选**。数据源 newsnow API 聚合 11+ 平台热榜(知乎/微博/财联社/华尔街见闻/B站/抖音/百度/澎湃/凤凰/头条/贴吧)+ RSS。

**核心能力**:

- 多平台热点聚合 + RSS + 关键词精准筛选(普通词/必须词/过滤词/正则)
- **AI 智能筛选新闻**(v6.5.0):自然语言描述兴趣,AI 提取标签 + 打分,`min_score` 阈值控制
- **MCP 架构 17+ 工具**:趋势分析(`analyze_topic_trend`)、情感分析(`analyze_sentiment`)、跨平台关联、智能摘要、时期对比
- 时间轴追踪:每条新闻首次→最后出现、排名变化、新增检测、持续性分析
- 9 渠道推送(微信/飞书/钉钉/TG/邮件/ntfy/Bark/Slack/Webhook)
- Docker/本地部署,数据本地自持(SQLite/S3)

**对腿 B 的价值**:这正是原方案"概念热度分 + 舆情情绪分"的**严肃解决方案**,把我之前判为"不推荐(反爬+水军噪声)"的股吧情绪因子**救了回来**:

- 反爬 → newsnow API 聚合,不用自维护 11 个爬虫
- 水军噪声 → AI 智能筛选 + `min_score` 阈值
- 结构化 → SQLite + MCP 工具能查趋势/情感
- **incremental 增量模式**专给投资者设计,零重复只推新增,适合实盘信号辅助

**局限**:

- newsnow 是热榜快照,**非每条新闻全文**;深度历史回测覆盖有限
- **冷门股/小盘股覆盖差**(热榜只推热门)
- 无结构化财务/事件数据(那是 FinanceMCP 的活)
- **GPL-3.0 copyleft**:VERA 个人实盘不分发,义务触发风险低;若未来商用分发需注意

#### 7.6.3 组合定位(关键结论)

| 工具 | 定位 | 腿 B 角色 | 数据类型 | 过拟合风险 |
|---|---|---|---|---|
| **FinanceMCP** | Tushare 结构化硬数据 | **地基** | 数值型(财务/资金/事件) | 低(直接进 selections) |
| **TrendRadar** | newsnow 舆情热度软数据 | **上层** | 文本型(情绪/热度) | 高(需 LLM 打分+验证) |

**两者互补不冲突**:一个结构化一个文本,一个 Tushare 源一个 newsnow 源,都能本地部署。

#### 7.6.4 腿 B 起步顺序最终版(覆盖 §7.3 旧建议 + §7.6.1 简化版)

考察两工具后,腿 B 起步定为**四档递进**(按过拟合风险 低→高):

1. **阶段 3a:FinanceMCP 结构化事件因子**(最先做)——龙虎榜/大宗/融资融券/资金流,数值型零 LLM,难过拟合,几乎零成本启动
2. **阶段 3b:TrendRadar 舆情热度因子**(中间)——情绪分/热度分,需 LLM 打分 + `min_score` 阈值,验证有效性后上
3. **阶段 3c:公告文本因子**(后续)——巨潮 cninfo + LLM,文本打分成本高
4. **阶段 3d:研报一致预期因子**(最后或放弃)——需 Wind 全文,Tushare 只有粗版

> 四档对应"过拟合风险 低→高"和"数据结构化 硬→软",从最不容易过拟合的开始。

### 7.7 文本/事件因子的前视偏差防护(HIGH,2026-07-19 审计补)

CLAUDE.md 历史审计铁律:2026-07-02 发现过前视偏差,系统综合 4.5/10。文本/事件因子是前视偏差高发地带,腿 B 一旦上必须严守:

**三条铁律**:

1. **事件对齐用公告日,非报告期**:年报"发布日 vs 报告期"必须用发布日对齐交易日;公告修正值不得回填历史;增减持用公告日(非权益登记日);ST 摘帽按历史状态逐日还原,不得用当前状态回填
2. **selections 过滤只用 t 日及之前已公告信息**:t+1 信息一律不得进入 t 日的选股过滤(这是未来函数)
3. **前视检测单测**:构造已知未来事件样本(如某股某日公告减持),验证因子值在公告日前为空、公告日及之后才出现

**典型陷阱清单**(腿 B 实现时逐条排查):

- 财报因子:用了报告期(如 2024Q1 报告期是 4 月,但年报次年 4 月才发)→ 必须用发布日
- 增减持:用了权益登记日(未来)而非公告日
- ST 状态:用当前是否 ST 回填历史(应逐日还原)
- 业绩预告:预告发布日 vs 业绩实际公告日
- 龙虎榜/大宗/盘后资金:T 日盘后数据,T+1 才能用(T 日收盘选股不能用 T 日龙虎榜,因为收盘时还没出)

这与信号日 T 收盘买入铁律一致:**T 日收盘时刻,只能用到该时刻已确知的信息**。

### 8.1 三道防线

- **阶段 2 启用 walk-forward**,不是阶段 1 就上(避免 MVP 过度复杂)

### 9.3 git 分支纪律(autoresearch 精髓)

```
autoresearch/<tag> 分支
  ├─ baseline commit (首次跑建立基线)
  ├─ keep commit A (fitness 提升,前进)
  ├─ keep commit B
  ├─ discard 实验 (reset 回退到 B,但 results.tsv 留痕)
  └─ ...无限循环
```

每次实验:
1. 当前 HEAD 上改公式/参数
2. git commit
3. 跑 `engine.run_cached` → grep fitness
4. fitness 提升 → 保留 commit(前进);下降 → `git reset --hard` 回退
5. results.tsv 记 `commit | fitness | 回撤 | 状态 | 描述`

**results.tsv 不入 git**(autoresearch 约定,留 untracked)。

### 9.4 并行会话 git 撞车防护(HIGH,2026-07-19 审计补)

MEMORY 记录 2026-07-18 亲历事件:并行会话会把未提交改动卷进它的 commit,HEAD 可能出半残 commit。循环夜里自动 `git reset --hard` + 无限 commit,若与用户在 master 上的工作交叉,极易:(1) 循环误把用户未提交改动 reset 掉;(2) 循环分支与 master 分叉后合并冲突;(3) "results.tsv 不入 git"与"核心源码 + tests 要及时入 git"边界模糊。

**三条防护**:

1. **循环跑前 git status 必须干净**:有未提交改动则 `git stash` 或跳过本轮,绝不盲目 reset
2. **物理隔离**:循环在单独 worktree 跑(`git worktree add ../vera-autoresearch autoresearch/<tag>`),不动主工作区
3. **results.tsv 写入 .gitignore**:明确 `data/autoresearch/*/results.tsv`,避免被批量 `git add` 误纳入
