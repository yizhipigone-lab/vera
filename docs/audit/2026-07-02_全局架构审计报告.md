## 0. 执行摘要（一页结论）

VERA 的「内核」——回测引擎、指标公式、阶梯止盈纯函数、ECharts 可视化骨架——设计上有亮点，但被「外壳失控」彻底拖累。06-25 审计指出的 5 大系统性缺陷，**7 天后无一根治**，反而暴露出一个比原问题更严重的新 CRITICAL：`engine.py` 用 `.bfill()` 把未来价格回填到停牌日，是教科书级前视偏差。

**当前最致命的判断**：回测绝对数字不可信，且不可信的方向是**系统性乐观**——零成本 + 未来价回填 + 跳空按止损线成交 + 涨停板能卖出，四重乐观偏差叠加。回测显示赚钱的策略，实盘可能亏钱。相对排序在 1d 同口径下尚可参考，但含停牌/ST 多的策略排序也开始失真。

## 2. CRITICAL 缺陷（9 项，必须立即处理）

### C1. `engine.py` 用 `.bfill()` 制造前视偏差【新发现，最严重】

- **位置**：`backtest/engine.py:368`、`:376`、`:385`
- **代码**：`close = close[cols].ffill().bfill()`（high_df、low_df 同样）
- **失效场景**：`bfill()` 用**未来**价格回填停牌/退市的 NaN。某股停牌 5 天，停牌期间的 close 被填为复牌当日收盘价 → 回测在停牌日"知道"了未来价。后果：
  1. 停牌日若触发止损/止盈，按复牌价成交（未来信息）。
  2. `_filter_limit_up`（engine.py:622）用 `prev_close = close.shift(1)`，前一日停牌 NaN 被 bfill 成当日价 → `prev_close == close` → 涨停检测失效，ST/涨停日买入信号漏过滤。
- **修复**：删除所有 `.bfill()`，仅保留 `.ffill()`；停牌日显式标记为不可交易（`tradable` 掩码），成交逻辑跳过停牌 bar。
- **核验**：审计师已直接读取 engine.py:360-391 确认三处 `.bfill()` 存在。

### C2. 默认 `enable_realistic_costs=False`，30+ 脚本仍在零成本回测

- **位置**：`backtest/engine.py:322`（默认 False）、`server.py:77/137`、`config/default.yaml`（未设）、30+ 批量脚本
- **失效场景**：C1 修复加了 `enable_realistic_costs` 开关，但默认关闭 → `eff_slippage=0.0`、`eff_stamp_tax=0.0`。经 server.py 跑回测，卖出只扣佣金万三，缺印花税 0.05% + 滑点 0.1%。单次买卖少扣约 0.15%，年换手 12 次 → 年化成本少扣 ~1.8%，直接抬高年化收益与夏普。全仓 grep 仅 `run_gs_txt_batch.py`、`tools/a2_recompute_equity.py`、`verify_critical_fixes.py` 三个脚本显式开启。
- **修复**：默认翻转为 `True`；`default.yaml:backtest` 段加 `enable_realistic_costs: true`、`stamp_tax: 0.0005`；`server.py:77/137` 同步。老脚本若需复现旧基线显式传 `False`。
- **核验**：审计师已读取 engine.py:315-339 确认默认 False 且 eff_slippage/eff_stamp_tax 置零。

### C4. 选股端 ST 过滤仍完全失效，`filter_stocks` 修复未接入

### C5. 复权口径不一致未接入断言

- **位置**：`core/formula_runner.py:29`（`dividend_type: int = 1`）vs `backtest/engine.py:356,592`（`"front"` str）vs `config/default.yaml:24`（`dividend_type: 1` int）

### C6. 跳空/跌停/涨停成交价乐观偏差（三重叠加）

- **位置**：`backtest/engine.py:153-156`（成本止损）、`:162-169`（阶梯止盈）、`:617-636`（仅过滤买入涨停，卖出侧无流动性检查）
- **失效场景**：
  1. **跳空跌停**：threshold=-0.12，但开盘直接 -20% 一字跌停。代码假设在 `ep*0.88`（-12%）成交，实际成交价 -20% 甚至根本卖不掉，每笔低估亏损 8 个百分点；连续跌停时完全无法卖出却按止损线记账。
  2. **阶梯止盈取最高档**：跳空高开同时触发 6%、15% 两档，代码把全部 30% 仓位都按 15% 档成交，实际应分档成交；日内冲高到 15% 又回落到 7% 收盘时，按 15% 成交是虚假成交价。
  3. ~~涨停板卖出限制~~ **（已撤回）**：原代理结论"涨停板卖不出"方向反了——涨停板上有大量买单封单，卖单秒成交，**卖得出**。真正的流动性问题只在跌停侧（买盘枯竭），已在子项 1 覆盖。此条作废。
- **修复**：跌停日成交价取 `min(stop_price, open_price)`；阶梯止盈按各档 profit 加权成交价或取 `min(ladder_profits[li], close)`。
- **核验**：审计师已读取 engine.py:153-169 确认成交价取止损线而非市场价。
- **讨论校准**：用户指出"hi_pp 触及 15% 按限价单 15% 成交"本身合理（盘中落单可成交）。代码取"已触发档位最高价"作为成交价，对跳空高开场景正确；唯一细微偏差是盘中慢涨路径下，低档（6%）仓位也按高档（15%）计价，偏乐观但影响有限（仅低档那部分仓位、差几个百分点），OHLC 无法区分跳空/慢涨，可接受。

### C8. `/api/run` 阻塞事件循环，进度反馈形同虚设

- **位置**：`server.py:192` `async def run_pipeline`（内部全是同步 CPU 密集代码，无 `await` 让出点）+ `web/index.html:560-571` 800ms 轮询 `/api/status`
- **失效场景**：FastAPI 单 worker 事件循环被整段回测占满，前端轮询请求全部排队，直到回测跑完才一次性返回。跑几十个公式/多年回测耗时 30 秒~几分钟时，进度条和 step 文字始终卡在"初始化"，用户以为系统卡死。
- **修复**：`run_pipeline` 改为同步路由（FastAPI 丢线程池），或用 `BackgroundTasks` + 真正的后台任务队列；`/api/status` 走独立线程实时返回。

### C9. 安全边界被打破：CORS 全开 + /output 暴露 + 无鉴权

- **位置**：`server.py:31`（`allow_origins=["*"]`）、`server.py:34`（`app.mount("/output", ...)` 暴露全量交易记录/HTML 报告）、`server.py:475`（`result_id` 未校验，路径遍历风险）
- **失效场景**：用户在浏览器访问任意恶意网页，该网页 JS 可 `fetch('http://127.0.0.1:8080/api/results')` 读取全部策略持仓与收益，或 `POST /api/run` 触发回测耗尽资源。`GET /api/results/..%2f..%2fconfig%2fdefault.yaml` 可能逃出 results 目录读取任意文件。**违反 MEMORY 中"策略/资金为高级机密"的保密边界**。
- **修复**：CORS 收紧为 `null` 或白名单；`/output` 静态挂载去掉或加鉴权中间件；`result_id` 用正则 `^\d{8}_\d{6}$` 校验；增加 token 或 origin 校验。
- **缓解因素**：默认 `host=127.0.0.1`（server.py:500）不对外网暴露，部分缓解；但本机恶意网页仍可攻击。

## 4. 维度深挖

### 4.1 设计缺陷（架构层）

1. **配置真相源名存实亡**：`config/default.yaml` 是唯一的真相源声明，但全项目仅 3 个入口（`server.py`、`pipeline.py:33`、`stop_config.py`）真读它，53 个根脚本里 13 个仍硬编码 `STOP_CONFIG`，30+ 个绕过配置。`load_stop_config()` 仅被 18 个脚本使用，属少数派。设计意图与落地完全脱节。

2. **修复模块"造好未通"模式**：06-25 后新建了 `core/stock_filter.py`、`core/dividend_type.py`、`backtest/stop_config.py` 三个归一化模块，设计本身正确，但全部未接入主管线（`selector.py` 未调 `filter_stocks`、`assert_consistent` 零调用方、`load_stop_config` 仅 18/53 用）。这是最浪费的修复方式——投入了设计，零收益。

3. **基准对比模块口径全错**（4/10）：H4 年化公式错 + H5 算术超额 + H6 策略扣费指数不扣 + 缺沪深300/中证500（默认指数篮子偏小盘成长，无法评估相对大盘价值的超额）。"超额收益"数字当前只适合看方向，绝对值无意义。

4. **私有 API 被事实公共化**：`_simulate_core_v3`（下划线表私有）被 5 个生产脚本直调，`tests/test_engine_run_path.py` 的存在恰恰证明它已是事实公共接口，但无文档承诺。签名一改全崩。

5. **53 个根脚本 DRY 违反严重**：`STOP_CONFIG` 字典 13+ 处手写、`ENGINE_CFG` 字典 5+ 处手写、TDX 函数 lambda 实现 4+ 处重写（而 `core/formula_runner.py` 已有正式实现）、`sys.path.insert` 样板 ~20 处。

### 4.2 功能/逻辑漏洞（回测可信度层）

除 9 个 CRITICAL 外，补充以下：

1. **`engine.py:56` trades 数组上限静默截断**：`max_trades = n_dates*n_stocks//4+1000`，超限静默丢弃记录，equity 不受影响但 trades_df 不完整 → win_rate/profit_factor 失真且无告警，是报告数据不一致的隐藏来源。

2. **`engine.py:237-265` 换股卖出绕过 T+1 检查**：逻辑上当前安全（旧仓 entry_idx≤i-1），但若未来 entry 信号能同日重复触发，存在当日买当日卖风险。

3. **`engine.py:178-201` 阶梯部分卖出后跳过同 bar 剩余仓位的其他止损检查**：剩余 70% 仓位当日同时满足成本止损不会被检查，要等下个 bar。

4. **`metrics.py:33-35` `trading_days` 死变量**：算了但年化分支用 `len(equity)`，与 H4 同根因。

5. **`benchmark.py` 1w 周期年化高估 4.8 倍**：`bars_per_day={"1w":1}` → periods_per_year=252，但 1w 应是 52 周/年。

6. **`engine.py:295-303` 期末清仓未扣卖出手续费**：最终权益略高估。

7. **`selection/result_writer.py:57-59` 新股/退市股取价静默 None**：NaN 污染下游排序统计。

### 4.3 用户体验/前端（上次未审维度）
