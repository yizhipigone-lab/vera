## CRITICAL 发现

### C1 — `metrics.py:67-68`：`e` 变量未定义，异常路径触发即崩

**文件**：[backtest/metrics.py:67-68](backtest/metrics.py#L67-L68)

```python
                except Exception:                    # ← 没有 "as e"
                    logger.warning("hold_days 计算失败: %s", e)  # ← NameError!
```

**根因**：`except Exception:` 没有 `as e`，变量 `e` 在此作用域中不存在。

**触发条件**：交易数据中含不可解析的日期字符串时（`pd.to_datetime` 抛异常）。

**为什么 302 个测试没发现**：此 except 块在正常数据下从未进入。异常路径零测试覆盖。

**影响**：把"静默跳过指标"升级成"metrics 计算直接崩溃"。

**修复**：`except Exception:` → `except Exception as e:`（加两个字符 `as`）

---

### C2 — `stop_config.py:112`：`get_logger` 未导入，兜底逻辑崩溃

**文件**：[backtest/stop_config.py:112](backtest/stop_config.py#L112)

```python
        except Exception:
            logger = get_logger(__name__)   # NameError: 'get_logger' 未定义
```

**根因**：文件顶部只有 `from typing import Any, Dict, Optional`（第 18 行），`get_logger` 从未导入。

**触发条件**：`config/default.yaml` 不可读或 `stop_loss` 节点缺失。

**影响**：比旧代码更糟——旧代码至少能返回兜底配置让程序继续跑，新代码在异常处理器自身崩溃。

**修复**：文件顶部加 `from utils.logger import get_logger`

---

### 两个 CRITICAL 的系统根因

**"修复静默吞异常时，在 except 处理器本身引入了 NameError"**。这是改动的系统性盲区：
- 异常路径在正常 CI 中从不触发 → 302 个测试全绿
- 改动者只关注了"把 pass 改成 logger"，没验证变量绑定和导入

## HIGH 发现

### H1 — `stop_config.py:117-118`：兜底 `trailing_stop` 参数与 `default.yaml` 不一致（预存问题）

| 参数 | 兜底 (stop_config.py) | default.yaml | 差异 |
|------|----------------------|-------------|------|
| `activation` | **0.08** (8%) | **0.035** (3.5%) | 2.3x |
| `drawdown` | **0.05** (5%) | **0.01** (1%) | 5x |

yaml 加载失败时用户静默得到宽松得多的拖尾止损配置，行为偏离且难察觉。注释仅标注 priority 对齐，未提 trailing_stop 值不同。

### H2 — `result_writer.py`：5 处内联 `import logging` 绕过项目日志工具

4 个 except 块内用 `import logging; logging.getLogger(__name__)`，与项目统一风格（`from utils.logger import get_logger`）不一致。若未来 `get_logger` 加入特殊配置，这些内联 logger 不会继承。

### H3 — `PipelineResult.__contains__` 无测试

新增方法无任何测试覆盖。虽然功能经手动验证正确（`main.py:52` 的 `"error" in result` 对 dict 和 PipelineResult 都工作），但缺少回归保护。

## 额外隐患

1. **异常路径测试系统性盲区**：两个 CRITICAL 都因异常路径零测试而漏过。建议建立异常注入测试模式。
2. **异常日志模式不统一**：本次出现三种模式（`exc_info=True` / 直接引用 `e` / `as e` 后引用）。建议在 CLAUDE.md 定铁律：`except Exception: pass` → 改为 `except Exception: logger.warning("msg", exc_info=True)`（用 exc_info 不引用 e，防 NameError）。
3. **PipelineResult 类型分裂**：成功返 `PipelineResult`，失败返 `dict`。`isinstance` 不可靠，维护者易出错。
4. **stop_config 兜底值腐烂风险**：兜底字典和 yaml 的无形契约无机器验证（H1 已验证漂移）。建议加测试对比。
