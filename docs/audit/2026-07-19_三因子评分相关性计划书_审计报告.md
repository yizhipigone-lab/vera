### 2.2 [HIGH H1] "零改动复用 gs_run_one.py" 与 per-trade 数据需求矛盾

计划书 §3 阶段 B 声称:"复用 gs_run_one.py:39 跑 UPN 公式 得到每笔信号:(stock_code, select_date) + 对应交易收益";§4 表写"gs_run_one.py 跑 UPN 基线 **不改**"。

**实际代码行为**(`tools/gs_run_one.py:110-122`):脚本 stdout 只输出**聚合 JSON**:`status / signals / stocks / trades(笔数) / cumret / annret / maxdd / sharpe / winrate / ladder6 / ladder15`。**没有 per-trade 收益明细,也没有 selections 落盘**。

而 BacktestEngine.run 的 `result["trades"]`(`backtest/engine.py:1120, 1268, 1291` 调 `_build_trades` 在 `:1332-1346`)是带 `stock_code / entry_date / exit_date / profit_pct / exit_reason` 的 DataFrame——这是 per-trade 数据的真实来源。

**矛盾后果**:要做相关性分析必须拿到每笔 trade 的 profit_pct,但"零改动复用 gs_run_one.py"只给你聚合数。要么:
- (a) 改 gs_run_one.py 加 `--dump-trades` 参数(违反"零改动"承诺);或
- (b) 新写 wrapper 脚本直接调 `StockSelector(sel_cfg).run()` + `BacktestEngine.run()`,把 selections + trades 落盘成 parquet/pickle 供阶段 C 消费。

(b) 才是真正"零侵入"的工程路径,但它意味着 **gs_run_one.py 不是真正的复用入口**,§4 表的"复用方式"列需重写。

**修复建议**:
1. §3 阶段 B 改成:"新写 `tools/run_upn_baseline.py`(或类似名)直接调 `StockSelector + BacktestEngine.run`,落盘 `data/upn_baseline/selections.parquet` + `trades.parquet`;**gs_run_one.py 不动**,只是参考它的调用模式"。
2. §4 表 `gs_run_one.py` 行的"复用方式"从"跑 UPN 基线"改成"调用模式参考(脚本本身不复用)"。
3. §6 交付物表把新 wrapper 脚本加进去。

### 3.3 [HIGH H3] ts_code 分批策略对 moneyflow 是次优选择 — 基于误用的"6000 坑"

计划书 §3 阶段 A:"拉取策略:按 ts_code 逐只拉(绕过 6000 条/次上限,冲烟已验证该坑)"+ §5 风险表"tushare 6000 截断 按 ts_code 分批拉 + 缓存"。

**机器校验"冲烟是否真验证过该坑"**:`tools/verify_tushare_events.py:51-62` + `tools/smoke_event_factors.py:125-127` 三个 endpoint 全部用 `pro.moneyflow(trade_date=...)` 按 trade_date 拉,当日返回行数(腿B报告 L22-26):

| 接口 | 单日行数(20240105) | 距 6000 上限 |
|---|---|---|
| moneyflow | 5090 | 余 910,未触顶 |
| top_inst | 650 | 余 5350 |
| block_trade | 133 | 余 5867 |

**三个 endpoint 单日全在 6000 上限以内**。"6000 坑"是 `share_float` 独有现象(腿B报告 L30 单日返回 6000 即触顶,L53 已诚实记录"share_float 参数语义不明"),与这三个 endpoint 无关。

**结合阶段 A 到 B 到 C 顺序的代价**:阶段 A 在阶段 B 跑出 UPN selections 之前就要拉数据,此时还不知道 UPN 选了哪些股,ts_code 必须**全市场 5090 只都拉**:

| 策略 | 1 年 API 调用数 | 单调用行数 |
|---|---|---|
| trade_date(推荐) | ~244 次 | 5090(满) |
| ts_code(计划书写法) | ~5090 次 | ~244 |

**ts_code 比 trade_date 多 20x API 调用**,且无 subsetting 收益。tushare moneyflow 通常 500 次/分限速,5090 次需 ~10 分钟,trade_date 仅 ~30 秒。

**修复建议**(任选其一,优先方案 A):
- **方案 A(改动小)**:§3 阶段 A 把拉取策略改成"按 trade_date 逐日拉(单日小于等于 5090 远低于 6000 上限,无需分批),缓存按月或按季落 parquet"。把"绕过 6000 上限"措辞从 §3 阶段 A 和 §5 风险表里删掉或改成"share_float 类 endpoint 才需要 ts_code 分批,本方案三因子均不需要"。
- **方案 B(更高效)**:调整阶段顺序为 B 到 A 到 C,阶段 B 先跑出 UPN selections(预计仅 100-500 只),阶段 A 只对选中的 stock_code 用 ts_code 拉(5090 到 ~200 次,最快)。此方案需在计划书里显式说明阶段顺序变更理由。

### 3.5 [MEDIUM M1] smoke 全局 SD 常量需重构为 per-signal T

## 4. 风险识别 — WARNING

### 4.1 零侵入(engine/selections 层)+ 前视防护 — PASS

§4 + §5 的零侵入承诺在 `backtest/engine.py`、`selection/selector.py`、`core/formula_runner.py` 三个模块上**确实成立**:计划书的新增文件全是 `tools/factor_*.py` 和 `tools/fetch_event_factors.py`,与生产路径解耦。前视防护 `trade_date 小于 T` 与冲烟一致(§3.1 已校验)。

### 4.2 [HIGH H2] signal-to-trade join 逻辑未定义

§3 阶段 C 说"给每笔信号算 total_score" + "对比每组的平均收益",但**没说怎么把信号匹配到交易**。

**实际数据结构**(`backtest/engine.py:1332-1346` `_build_trades` 返回的 trades_df):
- 有 `stock_code`、`entry_date`、`exit_date`、`profit_pct`、`exit_reason`
- **没有 select_date(信号日)**——信号日在 `selections` DataFrame 里,不在 trades_df

**未定义的边界场景**:

| 场景 | 实际可能 | 计划书未说怎么处理 |
|---|---|---|
| 一个信号 到 0 笔交易 | 涨停买不进 / 停牌 / 资金不足 | 信号是否计入分组分母?算缺失还是丢弃? |
| 一个信号 到 1 笔交易 | 正常情形 | 直接用这笔 profit_pct? |
| 一个信号 到 N 笔交易 | 同窗口内先卖后买再卖(止盈后再买) | 取首笔?取均值?加权? |
| 多个信号(同股不同日) 到 同一笔交易 | 信号日 T1 买,持有中 T2 又出信号 | trade 算给 T1 还是 T2? |

这些直接影响"高分组平均收益"的数字正确性。不做定义会让分析结果不可复现。

**修复建议**:§3 阶段 C 开头加"信号到交易匹配规则"小节,推荐默认规则:
- 按 (stock_code, entry_date) 匹配;entry_date 小于等于 T+3 视为该信号的成交(T 日收盘买,允许 T+1/T+2 滑点)
- 一个信号 到 多笔:取**首笔** profit_pct(保守口径,反映信号首仓质量)
- 一个信号 到 0 笔:计入分组分母,profit_pct 计 NaN,**单独报"信号成交率"指标**(高分组成交率是否也更高,本身是评分有效性的次级证据)
- 多信号 到 同一笔:归最早信号日

### 4.3 [LOW L2] 向量化承诺缺失

腿B报告 L54 已警告:"冲烟用 iterrows(数据量小可接受);**生产必须向量化**"。`smoke_event_factors.py:133-142` 用 `iterrows()` 构造测试样本。生产 selections 1 年可能 ~200-500 条 三因子 join,数据量上升一个量级。计划书 §6 交付物 `tools/factor_score.py` 未承诺向量化。

**修复建议**:§6 `tools/factor_score.py` 行加"实现用 pandas 向量化(map/groupby/merge,不用 iterrows),与腿B报告 L54 的生产向量化要求对齐"。

### 4.4 [LOW L4] "不联实盘"措辞与铁律错配

§5 风险表"不联实盘 纯分析,不产生交易信号"。项目铁律 1(CLAUDE.md 业务铁律)"宏观/地缘事件绝不联仓位调度"针对的是**宏观/地缘类**因子。三因子(资金流/龙虎榜/大宗)是 **stock-level 事件数据**,不属该铁律的字面管辖。

**修复建议**:§5 该行依据改成更贴切的"实盘脆弱期保护(CLAUDE.md 项目定位):纯分析无副作用,先证明有效性再考虑接入策略",避免误引铁律。

## 6. 可测试性 — WARNING

### 6.1 统计方法 code-grade 程序化 — PASS

附录统计方法说明:

| 方法 | 实现路径 | 可程序化 |
|---|---|---|
| 分组对比(平均收益/胜率/夏普/样本量) | `df.groupby("band")["profit_pct"].agg(["mean","count"])` 等 | 是 |

### 6.2 [MEDIUM M4] "正相关显著"决策阈值未量化

§3 阶段 D 决策表"正相关显著(高分组收益明显高于低分组)"——"明显高于"是定性表述。附录给了 `p 小于 0.05 视为显著`,但**没给效应量阈值**:

- p 小于 0.05 但均值差 0.1% 算"显著"吗?(大样本下 p 容易小于 0.05)
- 高分组 sharpe 要比低分组高多少才算"有效"?

实操时容易出现"p 小于 0.05 但经济意义微弱"被误判为有效的情形。

**修复建议**:§3 阶段 D 决策表第一行改成可量化双条件:**"正相关显著 = t 检验 p 小于 0.05 **且** 高分组平均收益 减去 低分组平均收益 大于等于 3%(年化对应绝对收益差阈值,实施时可调)"**。附录补一句"效应量阈值(均值差)由 §7 决策点 #5 实施时确认,默认 3%"。

### 6.3 决策树无歧义 — PASS

§3 阶段 D 三分支(正相关显著 / 无相关或负相关 / 样本不足)互斥且覆盖完整,每分支后续动作明确。
