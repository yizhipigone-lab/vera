## 🤔 我额外想到的隐患

1. **CLAUDE.md 失同步风险**：第 78 行说"7.5/10"，CHANGELOG 说"9.0/10"。新人会话读 CLAUDE.md 会得到**两个不同的可信数字**——CLAUDE.md 是 system-reminder 自动加载的，CHANGELOG 是文件读取才看。优先信 CLAUDE.md，可能误判现状。

2. **`_verify_references.py` 的覆盖盲区**：只校验 `[file.py:line]` 存在 + 行号范围。**不校验**报告里**不带行号**的"代码事实"、**描述与代码语义错位**（C1 假阳性那种）、**断言失败但行号存在**（如 `test_metrics_67_actual_code_has_as_e` 只能防"行号不存在"不能防"行号存在但语义错"）。

3. **`tests/conftest.py` + `tests/test_end_to_end_real_pipeline.py` 双 fixture 共存**：新会话读 conftest 会以为已统一收口，但实际旧 fixture 还在。

4. **`backtest/engine.py` 1190 行 + `server.py` 452 行 双双超 800 行红线**——CHANGELOG 自承"已知,未拆分 (本次范围外)"。本次 4 轮迭代反复给这些文件打补丁，**总有一天会因"行数太多改不动"而停滞**。

5. **测试质量与覆盖率混淆**：CHANGELOG 说"502 测试 → 9.0/10"，但 5 个 skip 全部来自端到端真实测试。**真实生产路径的测试覆盖 ≈ 0**，新会话会被"502 passed"误导以为回测路径已被全面守护。

6. **`test_benchmark_does_not_import_engine` 是空转测试**（T-CR-1）——即使 P2-6 修复**事实上正确**（grep 验证 benchmark.py 真不依赖 engine），但 CI 实际**未验证这一点**。如果未来有人写 `from backtest.engine import BacktestEngine` 加到 benchmark.py，这个测试还会 PASS，给"已验证"的虚假安全感。

7. **`assert_state_dtype` 测试自证自话**（T-CR-2）——函数本身只能验证"自己构造的合法对象合法"，根本不是守卫。**未来如果有人用 `Position(code=3.5)` 触发 dtype 错位 bug，这两个测试照样绿**，P1-3（CLAUDE.md 第 84 行说"核心函数必跑 tests"）的守护在 dtype 维度完全缺失。

8. **`stop_config.py:91-92` vs line 124 显示不一致**这种"显示文本与实际值漂移"是历史 H1 教训的复发——下次跑 `/api/strategy_summary` 接口用户看到的还是 `0.08/0.05`，但实际跑回测用 `0.035/0.01`，**口径漂移**会被用户发现为"为什么 UI 说的和回测结果对不上"。
