## 🤔 我额外想到的隐患

1. **`importlib.reload` 的真实用途 plan 没查根因**。`server.py:307-309` reload(stop_manager/metrics/engine) 三连击——这大概率是**开发期热重载**（改 engine.py 不重启 server 即可生效）。C1 直接删,开发体验（DX）会退化：以后改核心模块都得重启 FastAPI。plan 应说明"是否提供替代热重载方案（如 uvicorn --reload）",否则删了会埋怨。

2. **C3 把 22 处 tqcenter import 收口到 connector 1 处——收益大但单点风险也集中**。tqcenter 是通达信本地插件（`E:/NEW_TDX/PYPlugins/user/tqcenter.py`）,路径写死在 connector.initialize 的 sys.path 注入里。收口后,TDX 装到别的目录就只改 connector 一处（好事）,但 connector.initialize 失败时**所有** tq 调用全崩——需确认没有静默退化（accessor 失败应 fail-loud 抛 RuntimeError,plan 的 `ensure_connected` 已抛 ✅）。

3. **C1 ResultWriter 循环 import 陷阱**。`_default_status_sink` 用 `from server import pipeline_status` 动态 import 规避循环,但 server 又会 import ResultWriter。callable 注入 + 延迟 import 方案对,但**测试要覆盖"ResultWriter 先于 server 初始化"场景**,否则生产首次启动可能撞循环。

4. **数据量 10 倍时的隐患**：C4 把 `_gs_run_one` 改走 Pipeline.run——Pipeline 每次 run 都 `TdxConnector.initialize + close`。当前 5 个父 driver 用 subprocess 批量调（每公式一进程）,行为不变。但若未来有人改成 in-process 批量调,initialize/close 循环会有性能问题。plan 可加一句"Pipeline.run 不适合 in-process 高频复用,批量场景仍走 subprocess"。

5. **🆕 server 进度倒退（HIGH-6）若不随 C1 修,会一直留在线上**。即使 C1 最终落地,从现在到 C1 合并期间,线上 web 进度条仍会 40→30 跳动。建议把 `server.py:363-369` 这两行的修复作为独立的小 PR 先合（不依赖 C1）,别让现成 bug 等 C1。
