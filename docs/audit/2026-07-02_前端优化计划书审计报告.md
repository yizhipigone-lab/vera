## 1. CRITICAL — 必须修正才能执行

### C1. P0-2 基准线修复方案 B 有二次归一化错误

**计划书方案**（第 66-72 行）：

```javascript
// 找到基准首个有效数据的索引，只从该点开始画线
const firstValidIdx = bmVals.findIndex(v => v !== null);
if (firstValidIdx < 0) continue;  // 完全无数据则跳过
const bmAligned = bmVals.slice(firstValidIdx);
const bmDates = dates.slice(firstValidIdx);
// 归一化：从首个有效点起算为 0%
const bmStart = bmAligned[0];
const bmPct = bmAligned.map(v => v !== null ? v - bmStart : null);
```

**问题**：后端 `server.py:379` 返回的 `index_close` 已经是归一化值。源码 `index.html:649` 证实：

```javascript
bmMap[d] = (r.index_close - 1) * 100;  // 已经是百分比偏离
```

也就是说，`bmVals` 里的值已经是"相对基准首日的涨跌幅百分比"。如果科创 50 从 2020 年开始有数据，到 2024 年策略开始时已经涨了 30%，那 `bmVals` 在策略首日的值就是 `30`（代表 +30%），不是 `0`。

用 `v - bmStart` 做二次归一化，等于把基准在策略开始前的所有涨幅全部抹掉，强制设为 0%。这会导致：

- **用户看到"基准从 0% 开始画"**，而策略可能已经 +5%，对比完全失真
- **科创 50 等晚上市指数被人为压低**，与上证指数放在同一图里不可比
- 本来想修"断点"，结果修成了"错误起点"

**正确方案**：只裁掉开头无数据的 null 段，不做二次归一化：

```javascript
// 方案 B 修正版：裁 null 但保持原始归一化口径
const firstValidIdx = bmVals.findIndex(v => v !== null);
if (firstValidIdx < 0) continue;

// 策略线保持全量，基准线在无数据段填 null，有数据段保持原值
const bmData = new Array(firstValidIdx).fill(null).concat(bmVals.slice(firstValidIdx));

series.push({
  name: bmNames[name] || name,
  type: 'line',
  data: bmData,           // 与策略等长的 data 数组，xAxis 对齐
  lineStyle: { color: bmColors[ci % 4], width: 1, type: 'dashed' },
  symbol: 'none',
  connectNulls: true       // 跳过开头 null 段后连线
});
```

**影响**：如果照计划书方案执行，所有含基准对比的权益曲线图都会产生视觉错误，用户反而不如现状（断点但数值对）。

### C2. P1-2 "恢复默认"硬编码的默认值与后端三处打架

**计划书方案**（第 139-150 行）中前端硬编码默认值：

```javascript
cfgCondTimeProfit: '2',   // ← 计划书写 '2'（代表 2%）
```

**后端存在三处默认值互相矛盾**：

| 位置 | 代码 | 值 | 含义 |
|------|------|----|------|
| 前端 HTML `index.html:268` | `value="2"` | `2` | 输入框显示 2（代表 2%） |
| 后端 DEFAULTS `server.py:82` | `"cond_time_profit": 0.02` | `0.02` | 后端默认 2% |
| 后端 `get()` fallback `server.py:152` | `cfg.get("cond_time_profit", 0.01)` | `0.01` | 后端 fallback 1% |

**三处默认值互不一致**：

- 前端写 `2`（→ 传给后端 `2/100 = 0.02`）
- 后端 DEFAULTS 写 `0.02`（= 2%）
- 后端 `get()` 写 `0.01`（= 1%，与前两者差一倍）

这正是后端审计报告 CRITICAL #2 "引擎默认值与 yaml 不符"的前端镜像。如果用户点"恢复默认"，前端设 `cfgCondTimeProfit=2`，发给后端变成 `0.02`；但如果后端某处 fallback 到 `get()` 的默认值 `0.01`，实际执行的是 1% 而非 2%。

**影响**：不仅仅是 `cond_time_profit` 一个字段。计划书里硬编码的所有默认值（`cfgCostStopVal: '12'`、`cfgTrailingAct: '8'` 等）都需要逐一与后端 `server.py:77-84` 的 DEFAULTS 和 `server.py:148-153` 的 `get()` fallback 对齐。

**正确方案**（二选一）：

1. **调用后端 API**（推荐）：后端已有 `/api/config/defaults` 端点（`server.py:159`），前端从服务端拉默认值，不硬编码
2. **先对齐后端三处再硬编码**：把 `server.py:152` 的 `0.01` 改为 `0.02`，然后前端硬编码与 DEFAULTS 一致

### C3. P3-3 默认主题换 light 会导致暗色用户 FOUC + 主题无持久化

**计划书方案**（第 406-419 行）：

> `:root` 改为 light 色值，新增 `[data-theme="dark"]` 块

**现状分析**：

- `index.html:2` 硬编码 `data-theme="light"`
- `index.html:9-16` `:root` 写的是**暗色值**
- `index.html:17-22` `[data-theme="light"]` 覆盖为亮色
- `index.html:393-397` `toggleTheme()` 修改 `data-theme` 属性但**不写入 localStorage**

**问题 1：主题切换无持久化**

用户在 session A 切到暗色，刷新后 `index.html:2` 的 `data-theme="light"` 生效，又回到亮色。`toggleTheme` 只改了 DOM 属性，没调 `localStorage.setItem`。

**问题 2：换 `:root` 为 light 后的 FOUC**

当前 `:root` = 暗色，`data-theme="light"` + CSS 覆盖 = 亮色，所以老用户打开页面是：
1. HTML 解析 `data-theme="light"` → 应用 `:root` 暗色 → 白屏（0.1s）
2. CSS 引擎计算 `[data-theme="light"]` 覆盖 → 亮色显示

这个闪烁目前很短（<100ms），因为两套色值都在同一个 `<style>` 里。

但计划书改为 `:root = light`、`[data-theme="dark"]` 后，如果用户上次选了暗色（但没持久化），刷新后：
1. HTML `data-theme="light"` → `:root` 亮色 → 白色背景
2. 用户看到白色 → 手动切暗色
3. 再刷新 → 又白色 → 再切 → **每次刷新都闪白**

**修复**（两步）：

**第一步**：在 `toggleTheme` 中加入持久化：

```javascript
function toggleTheme() {
  const next = getTheme() === 'dark' ? 'light' : 'dark';
  document.documentElement.setAttribute('data-theme', next);
  localStorage.setItem('vera_theme', next);  // ← 新增
  themeIcon.innerHTML = next === 'dark' ? sunIcon : moonIcon;
  if (lastResult) renderAllCharts(lastResult);
}
```

**第二步**：在 `<head>` 最前面（CSS 之前）加内联 JS，消除 FOUC：

```html
<script>
  // 在 CSS 解析前设置主题，消除白屏闪烁
  (function() {
    var t = localStorage.getItem('vera_theme') || 'light';
    document.documentElement.setAttribute('data-theme', t);
  })();
</script>
```

然后才能安全地把 `:root` 改为 light 色值。

## 2. HIGH — 应修正再执行

### H1. P0-1 月度收益修复遗漏了跨年排序 bug

**计划书方案**本身正确（简单加总→复合收益），但源码 `index.html:693` 的排序有问题：

```javascript
const months = Object.keys(monthly).sort();
```

`Object.keys` 返回的 key 格式是 `"2024年01月"`、`"2024年10月"` 等。`sort()` 按字典序排列，结果：

```
2024年01月 → 2024年02月 → ... → 2024年09月 → 2024年10月 → 2024年11月 → 2024年12月
```

这恰好是对的（因为"年"、"月"的 UTF-8 编码不影响数字部分的字典序）。**但如果跨年**：

```
2024年10月 → 2024年11月 → 2024年12月 → 2025年01月 → 2025年02月
```

字典序下 `2024` < `2025`，跨年排序也是对的。**经核实，此项为计划书未提及但现有代码无 bug**。

然而，修复月度收益公式后，如果某月只有 1 天数据（如月末最后一个交易日），复合收益和简单加总差异极小；但如果某月有 20+ 天数据，差异可能达 0.5%+。**建议在计划书中补充一条验收标准**：对月内交易日 ≥10 天的月份，复合收益与简单加总的差值应 > 0。

### H2. P1-1 取消按钮的快照时机有 race condition

**计划书方案**在 `toggleEdit` 时快照 `data-original`。当前 `toggleEdit` 只做 `classList.add('editing')`（`index.html:478`），不涉及字段赋值。

**场景**：

1. 用户点"编辑" → `toggleEdit` 快照 `data-original` = 原始值 `12`
2. 用户改值为 `15`
3. 用户再次点"编辑"（同一 block，但代码没有防重复点击的 guard）
4. `toggleEdit` 再次快照 `data-original` = 已改的值 `15`（覆盖了原始值 `12`）
5. 用户点"取消" → 恢复到 `15` 而非 `12` → **取消无效**

**修复**：加 guard——只在首次编辑时快照：

```javascript
function toggleEdit(blockId) {
  const block = document.getElementById(blockId);
  block.querySelectorAll('.cfg-fields input, .cfg-fields select').forEach(el => {
    if (!el.dataset.original) {  // 只在首次编辑时快照
      el.dataset.original = el.type === 'checkbox' ? el.checked : el.value;
    }
  });
  block.classList.add('editing');
  block.querySelector('.cancel-btn').style.display = '';
}
```

同时，"编辑"按钮在编辑状态下应隐藏或变为"取消编辑"，避免重复点击。

### H5. P2-2 hex+alpha 后缀在 ECharts 中的兼容性

**计划书方案**（第 272-274 行）：

```javascript
const downShades = [c.down+'99', c.down+'bb', c.down];  // 如 '#1a7f3799'
const upShades = [c.up, c.up+'cc', c.up+'99'];           // 如 '#d1242fcc'
```

`getColors()` 返回的是 6 位 hex（如 `#d1242f`），加 `'99'` 后变成 8 位 hex `#d1242f99`。

**问题**：

- ECharts 在 SVG 渲染模式下不支持 8 位 hex alpha
- Canvas 渲染模式下部分浏览器不支持 8 位 hex
- 更安全的写法是用 `rgba()` 或 `color-mix()`

**修复**：封装一个辅助函数：

```javascript
function hexToRgba(hex, alpha) {
  const r = parseInt(hex.slice(1,3), 16);
  const g = parseInt(hex.slice(3,5), 16);
  const b = parseInt(hex.slice(5,7), 16);
  return `rgba(${r},${g},${b},${alpha})`;
}

const downShades = [hexToRgba(c.down, 0.6), hexToRgba(c.down, 0.73), c.down];
const upShades = [c.up, hexToRgba(c.up, 0.8), hexToRgba(c.up, 0.6)];
```

### M3. Google Fonts 超时降级不够完善

计划书写"内网无法访问则跳过"，但 `--font` 变量已声明 `'Inter', ...`。如果 `<link>` 标签加载但超时，会造成 3s+ 的 FCP 延迟（等待字体下载超时后才 fallback）。

**现状缓解**：Google Fonts URL 已自带 `display=swap`，不会阻塞渲染。但 `preconnect` 失败时 DNS 解析仍会消耗时间。

**建议**：加 `dns-prefetch` 作为最轻量降级，并在 `<link>` 上加 `media="print" onload="this.media='all'"` 实现非阻塞加载：

```html
<link rel="dns-prefetch" href="https://fonts.googleapis.com">
<link rel="preconnect" href="https://fonts.googleapis.com" crossorigin>
<link href="https://fonts.googleapis.com/css2?family=Inter:wght@400;500;600;700&family=JetBrains+Mono:wght@400;500&display=swap"
      rel="stylesheet" media="print" onload="this.media='all'">
```

## 5. 后端关联风险 — 最关键的审计意见

计划书**完全隔离于后端审计的 9 个 CRITICAL**，这是最大的系统性风险。以下是后端问题对前端计划的影响矩阵：

| 后端 CRITICAL | 对前端计划的影响 | 计划书是否考虑 | 严重性 |
|---|---|---|---|
| **#1 bfill() 前视偏差** `engine.py:368,376,385` | 前端展示的权益曲线本身就是基于前视数据计算的，修了月度收益公式还是错上加错 | ❌ | 🔴 |
| **#2 引擎默认值漂移** `engine.py:317,327` | P1-2 的"恢复默认"会把用户设成后端不认的值（本报告 C2） | ❌ | 🔴 |
| **#6 成交价乐观偏差** `engine.py:153-169` | KPI 展示的收益/夏普/回撤全是乐观的，修 UI 不修数据等于粉饰 | ❌ | 🔴 |
| **#7 核心代码未入 git** | 前端改动如果也不入 git，同样面临磁盘故障即丢的风险 | ❌ | 🟠 |
| **#8 /api/run 阻塞事件循环** `server.py:192` | 计划书 P1-3 改日志滚动，但进度轮询 800ms 在阻塞时形同虚设，用户看到的"运行中"长时间不更新 | ❌ | 🟠 |

**核心结论**：

> **在后端 CRITICAL #1（bfill 前视偏差）和 #6（成交价乐观偏差）未修复之前，P0 的"数据准确性修复"是伪命题。** 修了前端月度收益的复合算法，但底层数据本身就有前视偏差 + 乐观成交，用户看到的"修正后"数字反而比之前更"可信"——这不是修复，是**用精度掩盖准确度**。

**类比**：后端给你一把偏了 10° 的尺子，前端现在用这把尺子量得更精细了（从毫米级变成微米级），但角度偏了 10° 的事实没变。量得越精细，反而越让人误以为数据是准的。
