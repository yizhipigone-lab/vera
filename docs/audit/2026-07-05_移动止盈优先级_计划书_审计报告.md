## 🔴 一、逻辑漏洞

### L1. peak_hi 时序错位 — 触发检测用"上一根峰值",执行价算"当根 bar"

**位置**: plan §5.1.2 vs `engine.py:153-156`

```python
# engine.py:153-156 (现状)
if high_np is not None:
    if hi > pos_high_hi[p]:
        pos_high_hi[p] = hi              # ← 当根 bar 的 high 先写回 pos_high_hi
peak_hi = pos_high_hi[p] if pos_high_hi[p] > 0 else ep
```

- plan §5.1.2 写 `sell_price = peak_hi * (1.0 - trailing_drawdown)`,其中 `peak_hi` 来自 `pos_high_hi[p]`(153-156 行刚更新)。
- 触发检测在 212 行,而 153-156 已经在 212 之前执行了——所以 `peak_hi` 实际是**当根 bar 的 high**(已经包含当根)。
- 假设:前根 high=103,当根 high=104,当根 Low=99.5。回撤线 = `104×0.95 = 98.8`,Low=99.5 未触及 98.8 → 不触发。但用户期待"103 涨过 8% 后任何回落 5% 都该锁"。
- **相位错位的实际后果**:回撤线会比"真正的历史峰值"更紧,导致 trailing 触发频率**上升**(因为 peak 用了当根更高的 high),但**回撤线价也跟着抬**,可能错过本该锁利的 bar。
- **修正建议**:把触发检测处的 peak_hi 改为 `peak_hi_for_trail = max(pos_high_hi[p] before update, hi_for_this_bar)`,或者干脆**先触发检测、再更新 pos_high_hi**。

### L3. trailing 触发后没有"跳空低开"保护

**位置**: `engine.py:240-248` (cost_stop 的跳空保护模式)

```python
if triggered == 3:
    # 硬止损【简化模式】：使用stop_price执行
    # P1-1: 跌停保护 — 跳空低开时取 min(stop_price, open)
    stop_price = ep * (1.0 + cost_stop_threshold)
    if open_np is not None:

            stop_price = op
```

- cost_stop 触发后,如果开盘价已经低于止损线,按 Open 成交(更保守)。
- trailing 触发后,plan 直接按回撤线价成交,**没做等价保护**。
- 后果:开盘价已低于回撤线(比如一字跌停),按回撤线价成交会**高估**真实可成交价——回测结果偏乐观。
- **修正建议**:
```python
elif triggered in (4, 8):
    sell_price = peak_hi * (1.0 - trailing_drawdown)
    if open_np is not None:
        op = open_np[i, ci]
        if not np.isnan(op) and op < sell_price:
            sell_price = op   # 跳空低开保护
```

## 🟠 二、业务缺陷

### B1. 改了后端 default 改不了前端 HTML,用户看到 UI 与默认配置不一致

**位置**: `index.html:325-329` vs plan §5.4

- 前端 HTML 只 hardcode 2 个 radio,`default.yaml` 改成 `trailing_first` 之后:
  1. 第一次打开页面 → 勾选 ladder_tp_first(HTML 默认)
  2. 用户点"恢复默认配置" → 后端返回 `priority: "trailing_first"` → 前端 querySelector 找不到 → 跳过
  3. localStorage 里存了 trailing_first,但 UI 显示 ladder_tp_first
  4. 刷新页面 → loadConfig 走 154 行 querySelector 同样找不到 → trailing_first 被丢弃 → 又回 ladder_tp_first
- **整链路都不可用,但不会报错**,用户完全感知不到。
- **这是 plan §5.5 没真正做导致的核心问题**。

### B2. "止盈优先"摘要文案误导用户

**位置**: `vera-ui.js:303`

- trailing_first 不是"止盈优先",而是"移动止盈优先",含义与 ladder_tp_first 完全不同(后者强调阶梯止盈优先,前者强调盘中移动止盈优先)。
- 落到 "止损优先" 分支会让用户以为系统是"止损优先"模式,实际是"移动止盈优先"。
- **建议文案**:trailing_first → "移动止盈优先 (盘中锁利)";ladder_tp_first → "止盈优先 (阶梯锁利)";stop_first → "止损优先 (历史默认)"。

### B3. trailing_first 模式实际上常常退化为 ladder_tp_first

**位置**: `default.yaml:60-67` (默认 ladder 6%+15%,trailing 激活 8%)

- 默认 ladder 第一档 6% < trailing 激活 8%,所以当股价涨到 6% 时,ladder 已经触发并部分卖出(默认 30%)。
- trailing 激活需要 peak_hi_profit ≥ 8%,此时 ladder 第一档已经触发过(6%)。
- **trailing 实际很少有机会执行**——除非 ladder 档位都用完、继续上涨、回撤时才触发。
- 但默认 ladder_ratios=0.3+0.3=0.6,意味着 60% 已卖,**剩 40% 还可能被 trailing 锁利**。
- plan §4 表格 "ladder_tp > trailing" 是对的,但 plan 没点出**该模式在默认参数下,trailing 实际覆盖率很低**。
- **业务意义**:trailing_first 的"锁利"价值在默认配置下**主要靠 ladder 完成**,trailing 只是补 ladder 用完之后剩下的 40%。建议在 sumLadder 摘要里加一句"trailing 只对 ladder 档位用完后的剩余仓位生效"。

### B4. pp>0/pp<=0 不区分会让"亏了还在锁利"被记成"移动止盈"

**位置**: plan §3 第 3 行 + `engine.py:215`

```python
# plan §3: "pp>0/pp<=0 区分: 不区分, 整体优先"
# plan §5.1.1 伪代码:
if triggered < 0 and trailing_enabled and peak_hi_profit >= trailing_activation:
    trail_line = peak_hi * (1.0 - trailing_drawdown)
    if lo <= trail_line:
        triggered = 8 if pp > 0 else 4
```

- plan §3 写"不区分",但 §5.1.1 伪代码又区分 8/4——**plan 内部矛盾**。
- 即便按 §5.1.1 区分了,plan 仍没标:**pp<=0 时仍触发 trailing 这叫"移动止损"**,语义上用户期待"亏了别再锁利"。
- 实际业务:股价从 100 涨到 110(peak_hi_profit=10%≥8% 激活)→ 跌回 99(回撤=10%>5%)→ 触发 trailing,reason=4(因为 pp=(99-100)/100=-1% ≤ 0)。
- 听起来对,但如果用户把 trailing 当"锁利工具"用,看到自己"亏了还在'锁利'"会觉得策略在错误地卖。
- **建议**:reason 区分保留(8/4),但**显示层** (vera-ui.js 的 reasonDetail) 把"移动止盈" 文案改成"移动止盈 (回撤锁利)" 区分清楚。

### B5. trailing 触发模式不可选,intraday_low vs close_confirm 缺开关

### B6. 历史回测结果会与新语义不一致,但没版本标签

- **新语义**:回撤线价=101.97,但当天跌停=98,实际成交=98,**plan §2.3 的执行价 101.97 永远到不了**。
- 建议:trailing 触发后加"跌停保护",`sell_price = max(trail_line, low)`——保守取较高价(确保成交),并在 stop_price 处同步加 (engine.py:243-248 已有 cost_stop 的跌停保护模式)。

### F6. `trailing_drawdown = 0` 的边界

**位置**: `stop_manager.py` 初始化 + plan §2.2

## 七、一句话结论

**计划书的核心改造方向正确**(改 trailing 触发语义),但**至少 6 处严重问题必须改后才能开工**:

1. **前端 index.html 必须加第三个 radio**(否则后端 default 改了也无效,详见 B1)
2. **peak_hi 时序**要修正(否则盘中触及可能漏触发,详见 L1)
3. **trailing 跳空低开保护**要加(否则回撤线价高估,详见 L3)
4. **server.py fallback**要同步改(否则被静默回退,详见 L6)
5. **stop_manager best_reason**要加 trailing_first 分支(否则标签错乱,详见 L4)
6. **stop_config_summary 文案**要从 "Close回撤" 改成"盘中 Low 触及回撤线价成交"(详见 UI5)

**另外有 3 个隐藏业务风险**建议确认:

- **B3**:trailing_first 实际可能退化为 ladder_tp_first(因为 ladder 6% 在 trailing 8% 之前就触发)
- **B4**:pp>0/pp<=0 不区分会让"亏了还在锁利"的交易被记成"移动止盈",对账困惑
- **B5**:没有"收盘确认"开关,后悔药没有(plan §7.2 自己点了风险但没给方案)

**3 个待决策点**(D1/D2/D3/D4)需要用户拍板。
