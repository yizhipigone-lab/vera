### 🔴 CRITICAL-1：NaN 值直通服务器

| 属性 | 值 |
|------|-----|
| 编号 | VULN-2026-C1 |
| 严重性 | 🔴 CRITICAL |
| 位置 | vera-ui.js:304, 308, 317-334 |
| 类别 | 输入验证 |

**问题**：`runPipeline()` 仅校验日期格式和阶梯格式，不校验数字字段是否为有效数值。`parseFloat('')` 返回 `NaN`，`parseInt('')` 返回 `NaN`。当用户清空数字字段后点击"执行回测"：

- `initial_capital: parseFloat('')` → NaN
- `commission: parseFloat('')` → NaN
- `lot_size: parseInt('')` → NaN
- `max_hold_days: parseInt('')` → NaN

服务器端 Pydantic 模型使用 `Optional[float]`，NaN 会作为合法 float 值通过验证。下游回测引擎将以 NaN 参数运算，产生**静默损坏的结果**或深层崩溃。

**影响**：用户清空任意数字字段 → 发送 NaN → 回测引擎产生垃圾结果，无任何错误提示。

**修复建议**：
```javascript
// 在构建 config 对象前校验所有必需数字字段
const requiredNum = [
  ['cfgCapital', '初始资金'],
  ['cfgCommission', '手续费'],
  // ...
];
for (const [id, label] of requiredNum) {
  if (isNaN(parseFloat(document.getElementById(id).value))) {
    showToast(label + ' 必须填写有效数字', 'error');
    return;
  }
}
```

### 🔴 CRITICAL-2：XSS — innerHTML 未转义

| 属性 | 值 |
|------|-----|
| 编号 | VULN-2026-C2 |
| 严重性 | 🔴 CRITICAL |
| 位置 | vera-ui.js:639-665（交易表）、:255（日志）、:122-134（配置摘要） |
| 类别 | 安全 — 跨站脚本 |

**问题**：3 处使用 `innerHTML` 插入动态内容，均未转义：

1. **交易表**（line 639-665）：`t.stock_name`、`t.stock_code`、`t.exit_reason` 直接拼入 HTML
2. **日志面板**（line 255）：`msg`（含用户输入的 `formula_name`、服务器返回的 `data.error`）直接拼入 HTML
3. **配置摘要**（line 122-134）：`cfgLadderVal` 等用户输入值直接拼入 HTML

攻击路径：用户输入公式名 `<img src=x onerror=alert(document.cookie)>` → 日志面板渲染 → JS 执行。

**影响**：
- localhost 部署：风险较低（自攻击），但 localStorage 投毒可持久化 XSS
- 若暴露到网络：完全可利用

**修复建议**：
```javascript
function esc(s) {
  const d = document.createElement('div');
  d.textContent = s;
  return d.innerHTML;
}
// 所有 innerHTML 拼接处使用 esc()
```

### 🟠 HIGH-1：equity 零值与 null 混淆

| 属性 | 值 |
|------|-----|
| 编号 | VULN-2026-H1 |
| 严重性 | 🟠 HIGH |
| 位置 | vera-ui.js:415 |
| 类别 | 数据计算 |

**问题**：`const eq0 = data.equity[0].equity || 1` 使用 `||` 运算符，将 `0`（爆仓）与 `null`/`undefined` 等同处理。如果首日权益为 0（全部亏损），fallback 到 `1` 导致整条权益曲线计算错误——图表显示 `0%` 收益而非 `-100%`。

**修复建议**：`const eq0 = data.equity[0].equity != null ? data.equity[0].equity : 1;`

### 🟠 HIGH-2：hexToRgba 遇非 hex 颜色崩溃

| 属性 | 值 |
|------|-----|
| 编号 | VULN-2026-H2 |
| 严重性 | 🟠 HIGH |
| 位置 | vera-ui.js:2-7 |
| 类别 | 主题/兼容性 |

**问题**：`hexToRgba` 假设输入为 `#d1242f` 格式。但 `getComputedStyle` 可能返回 `rgb(209, 36, 47)` 格式（尤其是 CSS 变量值非 hex 设定时）。此时 `hex.slice(1,3)` 得到 `"gb"`，`parseInt("gb", 16)` = NaN，输出 `rgba(NaN, NaN, NaN, alpha)` — 无效颜色值。

**影响**：所有使用 `hexToRgba` 的图表元素（回撤区域、交易分布柱、dataZoom 滑块）可能透明/黑色。

**修复建议**：
```javascript
function hexToRgba(hex, alpha) {
  if (hex.startsWith('rgb')) {
    const m = hex.match(/(\d+)/g);
    return 'rgba(' + m[0] + ',' + m[1] + ',' + m[2] + ',' + alpha + ')';
  }
  const r = parseInt(hex.slice(1,3), 16);
  // ...
}
```

## 四、后端数据流验证

### 4.1 默认值不一致（报告 C2 — 已确认）

| 位置 | 字段 | 值 | 含义 |
|------|------|-----|------|
| server.py:82 DEFAULTS | `cond_time_profit` | `0.02` | 2% |
| server.py:152 get() fallback | `cond_time_profit` | `0.01` | 1% |
| index.html:307 | `cfgCondTimeProfit` | `"2"` | 2% |

**分析**：`StrategyConfig` 模型默认值为 `None`，实际运行时走 DEFAULTS 字典取 0.02，故 server.py:152 的 0.01 fallback 是**死代码**。但阅读代码的开发者会误认为默认值是 1%。这是一个语义陷阱。

### 4.2 恢复默认功能部分失效

`/api/config/defaults` 端点读取 `default.yaml`，但 YAML 中**没有** `cond_time_stop` 和 `first_day` 配置段。点击"恢复默认配置"时：

| 字段 | 是否被 API 重置 | 实际值来源 |
|------|-----------------|------------|
| cfgCondTimeDays | ❌ | HTML 默认值 "7" |
| cfgCondTimeProfit | ❌ | HTML 默认值 "2" |
| cfgFirstDayTarget | ❌ | HTML 默认值 "3" |
| cfgCondTimeEn | ❌ | 不被切换 |
| cfgFirstDayEn | ❌ | 不被切换 |

**影响**：恢复默认后，条件时间止损和首日规则的配置不会真正恢复。这是一个 P1-2 的**部分失效**。

### 4.3 基准字段名误导

API 返回的 `index_close` 字段实际是归一化比率（base-1），而非真实收盘价。任何第三方消费者看到 `index_close` 会误以为是原始价格数据。建议重命名为 `index_normalized` 或 `index_ratio`。

### 4.4 bfill 前视偏差 — 行号修正
