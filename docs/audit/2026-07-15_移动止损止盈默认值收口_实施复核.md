### 11.2 推荐下一轮：递归 JSON 安全序列化（T-H-1）

**选择理由（按独立性原则，单选一个）：**

1. **真正独立**：只动 `pipeline/result_writer.py` + `tests/test_safe_serialize.py`，不影响引擎、循环、Web 后端、HTML、CHANGELOG。
2. **真实风险**：FastAPI 的 `JSONResponse` 默认不允许 NaN/Inf（违反 RFC 8259）。当前 `safe_serialize` 只在 ResultWriter.serialize() 内部递归调用，但只在叶子节点替换 NaN/Inf 为 None。当某个 metrics 子字段是 dict-of-dict 或 list-of-dict 嵌套结构时，嵌套层级的 float NaN/Inf 不会被替换，会触发 API 层 500 错误。准确率复核 T-H-1 已最小复现。
3. **直接资金/可用性影响**：API 500 等同于"用户跑完回测但拿不到结果"，比覆盖率提升和共享 dict 更接近业务可用性。
4. **可测性强**：纯函数 + 嵌套结构 + NaN/Inf 边界，TDD 极友好。
5. **不阻塞其他 HIGH**：共享可变 dict、假测试、端到端集成测试、覆盖率到 80% 都是独立根因，本轮不塞进。
