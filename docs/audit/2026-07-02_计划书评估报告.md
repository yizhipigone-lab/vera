## 2. 计划书的硬伤（会导致返工或假修复）

### ❶ P0-2 漏掉 `run_cached` 的同款错误默认值 —— 批量优化路径仍会漂移

计划书只列了 `engine.py:317,327,423-425`。但 `run_cached()` 在 **engine.py:510-512** 有完全一样的错误默认值：

```python
cost.get("threshold", -0.08),          # 510
trail.get("activation", 0.05),         # 511
trail.get("drawdown", 0.03),           # 512
```

`run_cached` 是 18+ 个批量优化脚本的入口（审计自己说的）。只修 `run()` 不修 `run_cached`，等于"前台修好了，后门还开着"。

**这是 P0 级遗漏，必须补进 P0-2。**

### ❷ P0-1 删 bfill 但不碰 `fill_data=True` —— 修复可能部分失效

engine.py:356 调 `DataFetcher.get_kline(...)` 时**没传 `fill_data`**，而 data_fetcher.py:37 默认 `fill_data=True`，TDX 会在源头就填掉停牌 NaN。如果 TDX 的 fill 是前向填充，那么到 engine 这一层时根本没有 NaN 可 bfill —— **删 bfill 变成空操作**，前视偏差是否真消失取决于 TDX 的 fill 行为，不可控。

审计 H10 自己点了"回测取价统一 `fill_data=False`"，但**计划书 P0/P1 没有任何一项包含 `fill_data=False`**。

**正确做法**：P0-1 必须同时把 engine 的 `get_kline` 调用改为 `fill_data=False`，让停牌以 NaN 形式暴露，再由 `ffill()`（仅前向）做 mark-to-market，删 bfill 才有确定意义。否则就是"改了行号、没改行为"。

### ❸ P1-4 把 5m 年化的方向搞反了 —— 审计结论错，计划书照抄

审计 H4 说"5m 回测严重高估年化"，计划书 P1-4 验收标准写"5m/1w 回测年化不再高估"。但实测推导：

- metrics.py:40 用的是字面量 `252.0/len(equity)`，**不是** `periods_per_year`。
- 5m 一年 ≈ 48×252 = 12096 根 bar。bug 公式：`(1+cum)^(252/12096) = (1+cum)^0.0208` → 年化被压到接近零。**这是严重低估，不是高估。**
- 1w 一年 ≈ 52 根 bar。bug 公式：`(1+cum)^(252/52) = (1+cum)^4.84` → 确实高估 ~4.8 倍指数。✅

修法（用 `periods_per_year` 替换字面量 252）是对的。但**修完之后 5m 的年化会大幅上升、1w 的会大幅下降**——计划书写的"5m 不再高估"会让执行人误以为 5m 年化该降。方向反了，验收标准会误导。

此外 `metrics.py:33-35` 的 `trading_days` 是死变量（审计 4.2 也提了），P1-4 没顺手清。

### ❹ P1-6 只动 yaml 不动 `INDEX_CODES` —— 加了指数名但取不到数据

计划书说"默认指数篮子从 config 读取而非硬编码"+ "默认含 hs300/zz500"，只点了 `benchmark.py:21` 和 `default.yaml:64-69`。但真正的代码真相源是 **data_fetcher.py:17-22 的 `INDEX_CODES` 字典**，里面只有 shanghai/chuangyeban/kechuang50/zhongzhengA500，**没有 hs300、没有 zz500**。

`get_index_data` 的 fallback 是 `cls.INDEX_CODES.get(index_name, index_name)`（line 180）—— yaml 里写 "hs300"，TDX 收到的就是字符串 "hs300"，直接取数失败。

**P1-6 必须同时改 `data_fetcher.py:INDEX_CODES`，否则就是"配置加了个取不到的指数"。**

顺带：zhongzhengA500 映射的 `000510.SH` 是否真是中证A500代码也存疑，值得一并核对。

### ❺ P1-1 / P1-2 依赖 Open 列，但 engine 根本没取 Open —— 工作量严重低估

engine.py:356-364 只取了 `Close`/`High`/`Low`，**没有 `Open`**。而：

- P1-1 验收"跳空跌停按开盘价成交" → 需要 Open
- P1-2 验收"信号日次日开盘成交" → 需要 Open

data_fetcher 确实能返回 Open（data_fetcher.py:44-46 注释列出 Open），所以数据层 OK。但计划书 P1-1 没提"加取 Open 列、在 `_simulate_core_v3` 签名里加 `open_np` 参数、贯穿 high_np/low_np 同款链路"，3h/2h 工作量明显不够。

**更关键的是交互风险**：`_filter_limit_up`（engine.py:622）用 `close.shift(1)` 判涨停，买入价改 T+1 开盘后，entry_idx 语义整体后移一天，涨停过滤的 prev_close 基准也要跟着调整，否则又引入新的前视/漏过滤。这块计划书完全没提。

### ❻ P1-9 与现有设计冲突 —— "补扣卖出费"是 Hybrid，逻辑不自洽

engine.py:295 注释明确写"**期末不平仓，按市值计入**"。代码确实只做 mark-to-market，不卖。计划书 P1-9 说"期末清仓补扣卖出手续费"——但你既然没平仓，就没有卖出动作，扣什么费？这变成"既算市值、又扣一笔不会发生的卖出费"，会计口径混乱。

正确二选一：
- (a) 期末真实平仓、按 close 成交、扣卖出费；
- (b) 保持 mark-to-market、不扣费。

当前是 (b)，且 (b) 对"尺子量策略"是合理的（实盘也可能继续持有）。**P1-9 应该要么删掉、要么明确改成"期末平仓"并说明为何更优**，现在的写法是凭空多扣一笔。

## 3. 审计结论本身有问题、计划书照抄的项

### ❼ C3 对 server 路径的风险被夸大 —— server 有自己的正确默认层

审计说"server.py API 请求未传全字段，静默落到 -0.08/0.05/0.03/10万"。但 server.py:74-88 有一个独立的 `cfg.get()` 方法，里面 `DEFAULTS` 字典（line 76-84）的值是**对的**（100万/-0.12/0.08/0.05/2万）。`_config_to_yaml_dict` 里 line 135/142/148/149 那些错的 inline fallback（200000/10000/-0.08/0.05/0.03）是**死代码**，永远走不到。

所以 server 路径实际是安全的。C3 真正的风险面是**直接 `BacktestEngine({...})` 且不传全 stop_config 的根脚本**——那才会命中 engine 的错误默认值。计划书 P0-2 把 engine 默认值对齐 yaml 作"零成本兜底"是对的，但没必要在 server 上花功夫（server 已经对）。

### ❽ P0-5 引用的 `server.py:77,137` 是错的

server.py 既不读也不传 `enable_realistic_costs`（grep 确认全项目仅 3 处显式 True，server 不在其中）。翻转 engine 默认值为 True 后，server 因为不传该键 → 自动拿到 True，**问题自动解决**，无需改 server。

计划书写"server.py:77/137 同步"会让人去找一个根本不存在的字段，白费时间。真正要改的是 `default.yaml` 加 `enable_realistic_costs: true` + `stamp_tax: 0.0005`，加上 engine 默认翻转，server 路径就含成本了。

### ❾ benchmark.py 也有 1w 年化 bug —— P1-4 漏改

P1-4 只点了 `metrics.py:38-41` 和 `engine.py:313`。但 benchmark.py:42 `bars_per_day = {"1d":1,"1w":1,"5m":48}` 和 line 131 `periods_per_year = bars_per_day * 252` 是同一个 1w→252 的错误（应为 52）。超额收益年化（P1-5 几何化）和这个 1w 口径是连在一起的，必须一起改，否则 1w 周期的 benchmark 超额仍是错的。

## 5. 计划书没覆盖但该有的

1. **`_filter_limit_up` 与 T+1 开盘买入的耦合**（P1-2 改完，涨停过滤的 prev_close 基准要重算）——这是会悄悄引入新前视的点。
2. **`trades` 数组扩容（P1-8）的实现成本**：numpy 数组在循环中途扩容要么预分配翻倍、要么改 list-of-records，"1h + warning"低估了。建议改 `list` append + 末尾 `np.array`，比动态扩容 numpy 简单。
3. **回归基线缺一个数值锚**：P0 验证说"对比修复前基线"，但 `analyze_gupiao012.py` 是未跟踪文件，历史数字存在哪、口径是否含成本没说清。建议先固化一个"修复前快照"（commit + 跑一次存 JSON），否则改完没有可比基准。
