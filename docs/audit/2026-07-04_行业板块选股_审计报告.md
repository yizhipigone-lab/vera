#### ⚠️ 逻辑漏洞

##### 漏洞 #1 — `_SECTOR_CACHE` 共享引用 + `.clear()` 不一致（重要度：中）

- **代码位置**：[core/data_fetcher.py:208-209](core/data_fetcher.py#L208-L209) `clear()` + [core/data_fetcher.py:219-228](core/data_fetcher.py#L219-L228) `if cls._SECTOR_CACHE: return cls._SECTOR_CACHE`
- **问题分析**：
  ```python
  _SECTOR_CACHE: List[dict] = []  # 类变量，list 对象唯一
  if cls._SECTOR_CACHE:              # 判断当前是否空
      return cls._SECTOR_CACHE       # 返回同一个引用
  cls._SECTOR_CACHE = [...]          # 重新赋值（新 list 对象）
  return cls._SECTOR_CACHE
  ```
  `cls._SECTOR_CACHE.clear()`（clear_sector_cache）确实清空了。但**如果调用方把返回值存到了自己变量里**，下次 `clear()` 时**调用方的引用**也会被清空（list 是按引用传递）。
- **实际影响**：当前用法是 selector 里 `stocks.extend(sector_stocks)` 立即消费，不留引用。**理论上没问题，实际也无 bug**。
- **行业标准视角**：金融级应该 `return list(cls._SECTOR_CACHE)` 做防御性拷贝，避免调用方误改共享数据。
- **修复建议**：把返回都改成 `return list(cls._SECTOR_CACHE)` 或直接冻结类型。
- **实际阻断价值**：低。可记入技术债，但本次不强制修。

##### 漏洞 #2 — `exclude_new_listings_days: 60` yaml 默认被静默忽略（重要度：中-高）

- **代码位置**：[selection/selector.py:102-105](selection/selector.py#L102-L105)
  ```python
  exclude_new = u.get("exclude_new_listings_days", 0)
  if exclude_new > 0:
      logger.warning(f"exclude_new_listings_days={exclude_new} "
                     "— TDX get_stock_list 暂不支持按上市天数过滤，此选项被忽略")
  ```
- **问题**：[config/default.yaml:22](config/default.yaml#L22) `exclude_new_listings_days: 60`，**用户配置永远不生效**但又提示支持，让人误以为生效。
- **追溯来源**：这是 v3.3/v3.4 之前就存在的问题，**与本次"行业板块"无关**。
- **审计判定**：本次不要修，但要在审计报告里立 flag — 用户如果用 sectors 选新股活跃的板块（"半导体"里有大量上市 1 年内的新股），次新股过滤又无效，结果与预期不符。

##### 漏洞 #3 — `resolve_universe` 中 `etf_only` 与 `include_etf` 优先级日志自相矛盾（重要度：低）

- **代码位置**：[selection/selector.py:107](selection/selector.py#L107)

#### 🐛 BUG

##### BUG #1 — `selector.py:57` 中 `bool(u.get("etf_only", False))` 对 `u.get("include_etf", ...))` 的潜在 NULL 处理（重要度：低）

- **代码**：

##### BUG #2 — `_SECTOR_STOCKS_CACHE` 缓存值可能是空列表但仍视为"已缓存"（重要度：中）

- **代码位置**：[core/data_fetcher.py:237-251](core/data_fetcher.py#L237-L251)

      return cls._SECTOR_STOCKS_CACHE[sector_code]

- **结论**：代码目前是**只在成功路径写入缓存**，有检查。但需要再加一个测试覆盖这个 case。

## 第三部分：自检 — 我从代码里找出的 4 个逻辑漏洞（不在计划书里）

### 漏洞 #1 — `loadConfig` 不读 sectors 状态，但 `loadSectors` 顺序可能错乱

- **位置**：[vera-ui.js:908-909](web/vera-ui.js#L908-L909)
  ```javascript
  loadConfig();        // 同步, 读普通配置
  loadSectors();        // 异步, 调 /api/sectors
  ```
- **问题**：用户刷新页面时,`loadConfig` 没处理 sectors（sectors 是单独的 localStorage key `vera_selected_sectors`）。`loadSectors` 异步获取板块列表后,读 `localStorage.vera_selected_sectors` 渲染 checkbox 选中。**这是对的**。
- **风险**：如果用户开了 2 个 Tab，且同步没开，`/api/sectors` 在 Tab A 已缓存但 Tab B 第一次进，**同时访问 TDX**。**两个独立 TDX 连接可能冲突**。
- **判定**：VERA 是本地单用户系统，不构成问题。

### 漏洞 #2 — `loadSectors` 失败后 checkbox 区域永远显示错误（无重试按钮）

- **位置**：[vera-ui.js:154-165](web/vera-ui.js#L154-L165)
- **问题**：失败时把"sectorEmpty"内容改成错误文本,**没有暴露"重试按钮"给用户**。计划书 A2 明确要求"前端显示重试按钮"。
- **修复建议**：在 `sectorEmpty` 元素里追加 `<button onclick="loadSectors()">重试</button>`。
- **阻断价值**：中（用户体验，但用户多半会 F5）。

### 漏洞 #3 — 板块列表未加载完就勾选（异步竞态）

- **位置**：[vera-ui.js:909](web/vera-ui.js#L909)
- **场景**：用户快手 F5 后立刻勾选，但 `/api/sectors` 还没响应。
- **当前行为**：`renderSectors` 只在 `loadSectors` 拿到结果后调用，**checkbox 区域是"加载中..."文本**，用户根本勾不到。`onSectorToggle` 不会被触发。
- **结论**：**理论上没事**，因为 UI 在加载完成前没有可交互元素。
- **边界 case 升级**：用户点"清空"按钮 → 此时 `_selectedSectors=[]`（空） + `loadSectors` 还没回调 → `_allSectors=[]` → `renderSectors` 看到空返回"sectorEmpty"显示"无板块数据"。
- **判定**：用户操作可能不达预期，但不会出 bug。

### 漏洞 #4 — `clear_sector_cache` 在 `backtest/engine.py` 触及模块 import 缓存场景下可能无效

- **位置**：[server.py:251-254](server.py#L251-L254) 的 `importlib.reload(backtest.engine)` 等强制 reload。
- **问题**：每次 `runPipeline` 都会 reload `backtest.engine` 等模块，但 `core.data_fetcher` 不在 reload 列表里。`_SECTOR_CACHE` 持久生效。
- **结论**：这是预期行为（避免每次跑回测都重拉 128 个板块），不是 bug。
- **风险**：如果用户在回测中途板块名变了（理论不大可能），缓存过期。需要 `clear_sector_cache()` 手动操作。

## 附录 A：本次审计未触及但值得后续关注的边界

1. **TDX `tq.get_stock_list_in_sector` 的 list_type=0 参数**：计划书写 list_type=0，实际也用 0，但 TDX 官方文档这参数取值含义不明。如果 TDX 升级换参数名，会断。**风险低，影响大**。
2. **`_SECTOR_STOCKS_CACHE` 没加 LRU/TTL**：板块代码失效（公司退市后被剔除板块）会拿不到新数据，但用户一般不会察觉。手动 `clear_sector_cache()`。
3. **板块名称稳定性**：依赖 TDX 的中文板块名。如果 TDX 把"半导体"改名为"半导体制造"，前端搜索筛选会失效。需要对板块 name 做 hash 缓存。
4. **大并发场景**：未来如果做成多用户版，`_SECTOR_CACHE` 类变量不能多用户共享。**当前本地单用户不受影响**。
