## 三、C1 架构说明

### 旧架构（~2026-07-13）

```
server.py /api/run
  → 5 个直调 import（BacktestEngine / StockSelector / ...）
  → 手工编排（选股→数据→回测→报告）
  → 手工 persist(last.json)
  → 进度手工赋值
```

**问题**：
- 手工编排与 `Pipeline.run` 两条路径，行为可能漂移
- 进度回调不一致（`pipeline_status.progress` 手工写 vs Pipeline 回调）
- 250 行 handler 无法维护

### 新架构（2026-07-14 起）

```
server.py /api/run
  → _config_to_yaml_dict(cfg)
  → 写临时 YAML
  → Pipeline(tmp_yaml).run(progress_callback=ResultWriter.on_progress)
  → ResultWriter.serialize(result)  → 前端 JSON
  → ResultWriter.persist(...)       → last.json + results/ + index.json
```

**三个核心组件**：
- **`Pipeline.run()`**：唯一完整流程接缝（选股→数据→回测→基准→报告），返回 `PipelineResult`
- **`PipelineResult`**：frozen dataclass，dict-like 访问（`result.get("backtest")`），统一返回结构
- **`ResultWriter`**：`serialize()`（→前端）/ `persist()`（→三文件）/ `on_progress()`（回调适配器）

### C3 TdxConnector.tq() 说明

旧：各模块直接 `from tqcenter import tq`
新：`tq = TdxConnector.tq()` — 懒加载，统一出口

**好处**：
- 根目录只有 `connector.py` 一处需要知道 TQ 路径
- `TdxConnector.ensure_connected()` 集中管理连接生命周期
- 重连逻辑只写一处

### C2 StopManager 重复逻辑删除说明

**删了什么**：`engine.run()` 里有一段 label-enhancement 块，会把 `reason=1.0("换股卖出")` 的交易用 stop_manager 的 triple-match 重新计算 exit_reason 并覆盖。

**为什么删**：
- `_build_trades` 的 `reason_map` 已经是 exit_reason 的真相源
- stop_manager 重复计算会导致"换股卖出"被错误回填为"止损原因"
- 两套逻辑并存是语义冗余，不是功能增强

**影响**：trades 表的 exit_reason 分布会变（不再有 stop 回填换股卖出），但 pnl / metrics 完全不受影响。

> **用户须知**：若与历史 trades 数据对比发现 `exit_reason="换股卖出"` 的笔数变多、具体止损原因（如"成本止损"/"移动止盈"）变少，属正常现象——这是因为 C2 删除了原本把"换股卖出"回填为止损原因的重复逻辑，exit_reason 现在以 `_build_trades` 的 reason_map 为唯一真相源。
