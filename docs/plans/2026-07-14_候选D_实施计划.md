# 候选 D：架构全面收编（C1+C2 修正版+C3+C4）

## Context（为什么改）

候选 A 阶段 1（2026-07-13，commit 980b04f 等）已经把 `engine.run_cached` 加厚成全能入口、`_simulate_core_v3` 锁回私有。但 Explore agent（2026-07-14）巡出 18 个剩余架构摩擦点，按"两条平行实现互相漂移"主题筛出 7 项候选；用户挑了 4 项 Strong 一起做：

- **C1**：server.py `/api/run` 270 行 handler 完整重写管线，不调用 `Pipeline.run`
- **C2**：StopManager 路径里 `compute_exit_signals` / `_compute_single_stock` 与 `_simulate_core_v3` 平行（147 行重算止损决策）
- **C3**：`core/formula_runner.py` 与 `core/data_fetcher.py` 各 `from tqcenter import tq`（2 处 + 8 处）、各起 `_ensure_ready`、各调 `dividend_type` 边界归一
- **C4**：45 个根目录 `.py`（5,908 行）形成"影子管线"，3 类分流

**预期产出**：净减 600+ 行（保守预估 2800 行）；消除 4 处"必须手同步"漂移点；把 `Pipeline.run` 钉成**完整回测流程**（选股+回测+基准+报告）的唯一接缝——`server` / `main` / `_gs_run_one` 三入口统一走 Pipeline。**两层接缝边界**（2026-07-14 审计修正）：只跑回测核心或需多次对比的批量脚本（`backtest_quantqq_ab_compare` 跑两次回测、`backtest_quantqq_2026` 直调 FormulaRunner 绕过 StockSelector、`_run_sector_batch_v2` 纯批量）**保持直调 BacktestEngine**，等候选 B 的 BacktestRunner 抽象统一收编——它们走 Pipeline 会破坏功能或引入口径分裂（详见末尾"接缝边界说明"）。

## 关键事实修正（C2 范围调整的依据）

`engine.py:823,826,850, 981,984,995` 把 `StopManager` 真用：

| 位置 | 用途 |

class ResultWriter:

    def __init__(self, status_sink: Optional[Callable[[str, int], None]] = None):
        # status_sink 默认往 server.pipeline_status 写

        """Pipeline.run progress_callback adapter; 失败被吞。"""

    def persist(self, response: dict, *, results_dir: Path, last_result_path: Path,
                meta_extras: Optional[dict] = None) -> None:
        """复刻 server.py:496-528 — output/results/{ts}.json + index.json 50 条 + last_result.json.
        失败被吞（与 server 现状一致）。"""
        ...

def _default_status_sink(step: str, pct: int) -> None:
    """写 server 的 module-level pipeline_status 单例。通过 callable 注入避免循环 import。"""
    from server import pipeline_status
    pipeline_status.step = step
    pipeline_status.progress = pct

def _safe_name_map() -> dict:

        from core.data_fetcher import DataFetcher

## 接缝边界说明（2026-07-14 审计修正，HIGH-2 决策落定 = 走 A）

本 plan 原文"把 `Pipeline.run` 钉成唯一的'怎么跑一次回测'接缝"措辞过宽。经核实 defer 的 3 个直调 driver 的真实结构后，精确化为**两层接缝**：

### 第一层：完整流程接缝 = `Pipeline.run`（候选 D C1 达成）
`Pipeline.run` 是"选股 → 回测 → 基准对比 → 报告"的完整业务流程。以下入口走它：
- `main.py`（CLI 入口，已走 Pipeline ✅）
- `server.py /api/run`（C1 将改为走 Pipeline）
- `tools/gs_run_one.py`（C4 将迁入并改走 Pipeline）

C1 完成后，"完整流程走 Pipeline"这条接缝统一。

### 第二层：回测核心接缝 = `BacktestRunner`（候选 B 阶段 2，**不在本 plan**）
以下脚本**不该走 `Pipeline.run`**，保持直调 BacktestEngine，等候选 B 的 BacktestRunner 抽象统一收编：

| 脚本 | 为什么不走 Pipeline.run（核实证据） |
|---|---|
| `backtest_quantqq_ab_compare.py` | A/B 对比需**跑两次回测**（同信号、两组 priority，:63-77），`Pipeline.run` 只跑一次。硬塞破坏核心功能 |
| `backtest_quantqq_2026.py` | 选股直调 `FormulaRunner.run_stock_selection_with_dates`（:42，绕过 `StockSelector.resolve_universe` 的 ST/涨停过滤）。塞进 Pipeline 会**改变选股口径**，违反口径一致性铁律 |
| `_run_sector_batch_v2.py` | 纯批量回测，无选股概念（grep `StockSelector/Pipeline` 0 命中）。走完整流程纯属强加 |

**结论**：候选 D 的"唯一接缝"目标 = **第一层达成**。第二层是候选 B 的活，本 plan 不碰。这不是缩水，是 `ab_compare` 的结构（跑两次）和 `quantqq_2026` 的口径独立性决定的**架构正确**——方案 B（硬塞进 Pipeline）会损坏 ab_compare 功能 + 引入口径分裂 bug，已否决。
